<objective>
Add a "Reset Progress" control to the title screen that clears the game's saved
progress (the `mathlab_platformer_v2` localStorage save) so the player can start
completely fresh, gated behind an explicit, deliberate confirmation step so it
can never be triggered by a stray click or keypress.

Purpose: the 12-year-old target player currently has no way to wipe her save
short of manually opening browser devtools. A discoverable but safely-gated
in-game control closes that gap without risking an accidental wipe.

Output: `resetSave()` in `src/progress.js` (guarded persistence seam, mirrors
`loadSave`/`writeSave`); a "press R to reset progress" prompt + Y/N confirm
overlay on the title screen (`src/scenes/title.js`); new layout constants in
`src/config.js`; an updated structural gate in `scripts/check-progress.sh`.
</objective>

- The save key is `CONFIG.SAVE.KEY` = `"mathlab_platformer_v2"` (src/config.js:164).
  There is no separate AUD-03 mute-toggle key anywhere in the codebase yet
  (confirmed via `grep -rn localStorage src/` — the only key in use is
  CONFIG.SAVE.KEY) — so the "must not touch AUD-03's key" constraint is
  satisfied simply by never using a wildcard clear (`localStorage.clear()`),
  only a named `removeItem(CONFIG.SAVE.KEY)`.
- Kaplay's global `onClick(cb)` (used by title.js's existing "click anywhere to
  start" behavior) is a raw, unscoped mouse-press listener — it fires on ANY
  click anywhere on the canvas, with no concept of "topmost object" or z-order
  occlusion (confirmed by reading the vendored `lib/kaplay.mjs`: the 1-arg form
  of `onClick` calls `a.app.onMousePress(t)` directly). A mouse-clickable
  "Reset Progress" button would therefore ALSO fire the existing
  "start"->go("select") handler on the SAME click — both listeners see the same
  mouse-press event. This is why the reset control below is a KEYBOARD trigger
  ("R"), never a clickable box: a click-based trigger cannot be made safe
  against this collision.
- `scripts/check-progress.sh` is the project's structural verification gate for
  the whole persistence layer (no JS test framework in this project — grep-based
  assertions ARE the tests). `scripts/smoke-progress.mjs` is the pure-math node
  smoke it invokes at the end; it is NOT the place to test `resetSave()` (that
  function touches storage, and smoke-progress.mjs deliberately never touches
  storage — same reason `loadSave`/`writeSave` aren't exercised there either).
