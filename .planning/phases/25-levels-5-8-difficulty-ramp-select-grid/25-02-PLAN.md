<objective>
Fix both Playwright audit scripts' select-screen navigation loops (25-RESEARCH.md Pitfall 1) so they compute a row/col position instead of assuming a single flat row of tiles. This is a Wave 0 requirement from 25-VALIDATION.md: without this fix, neither script can ever reach levels 5-8 once `select.js` grows to a row-wrapped 2x4 grid, silently breaking this phase's own interactive-audit proof and any later regression run.

Per 25-RESEARCH.md's Open Question 1 resolution: fix the select-nav loop in THIS phase (cheap, and otherwise blocks even a basic regression smoke of levels 5-8) but leave the full, formal, retry-hardened multi-attempt 8-level closure sign-off to Phase 28 (VALID-03) -- this plan does NOT touch `scripts/lib/audit-retry.mjs`, does NOT add new retry infrastructure, and does NOT change `scripts/lib/mechanic-drive.mjs`.

Purpose: `scripts/browser-boot.mjs:134-139` and `scripts/audit-phase21-mechanics.mjs:166-176,190-199` both press `ArrowRight` exactly `i` times from a fresh select-screen entry -- a flat, row-blind loop that will silently never leave row 1 once row-wrapped Left/Right ships (Plan 25-05's locked decision: "wrapping within a row only, no cross-edge wrap"). Per the project's documented convention (STATE.md: duplicated Playwright code is fixed "by hand in each copy," not extracted to a shared module), this plan applies the identical fix independently at all THREE call sites (one in browser-boot.mjs, two in audit-phase21-mechanics.mjs).
Output: `scripts/browser-boot.mjs`'s per-level loop and `scripts/audit-phase21-mechanics.mjs`'s two call sites (its main per-level loop and its `reloadLevel` callback) each compute `row = Math.floor(i / 4)` / `col = i % 4`, then press `ArrowDown` `row` times followed by `ArrowRight` `col` times, assuming the select screen's fixed 4-column layout.
</objective>
