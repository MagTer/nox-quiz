    CombatInputHandler structure:

    Private field: `locked: false` — blocks re-entrant answers during feedback pause
    Private field: `currentQuestion: null` — holds the active question object from QuestionSelector

    Method `setup()`:
    - Attaches a change event listener to `document.getElementById('combat-question')` (the combat fieldset from Plan 01)
    - Guard order inside the listener: (1) `if (App.mode !== 'dungeon') return;`, (2) `if (CombatInputHandler.locked) return;`, (3) `if (e.target.tagName !== 'INPUT' || e.target.type !== 'radio') return;`

    Method `handleAnswer(selectedValue)`:
    - Sets `CombatInputHandler.locked = true`
    - Parses the value: `const selected = parseInt(selectedValue, 10);`
    - Gets the current question: `const q = CombatInputHandler.currentQuestion;`
    - Determines correctness: `const isCorrect = (selected === q.answer);`
    - Calls `CombatEngine.resolveAnswer(isCorrect)` and stores result as `const result`
    - Updates HP bars: `DungeonRenderer.updateHP(result.enemyHP, result.playerHP);`
    - Shows floating damage number:
      - If isCorrect: `DungeonRenderer.showDamageNumber(document.getElementById('combat-enemy-area'), CONFIG.DUNGEON.DAMAGE_CORRECT, true);`
      - If not isCorrect: `DungeonRenderer.showDamageNumber(document.getElementById('player-hp-container'), CONFIG.DUNGEON.DAMAGE_WRONG, false);`
    - Shows RPG feedback text:
      - If isCorrect: `DungeonRenderer.showFeedback(Math.random() < 0.2 ? 'Critical hit!' : 'Attack!');`
      - If not isCorrect: `DungeonRenderer.showFeedback('You took a hit!');`
    - If result.leveledUp: call `Renderer.showLevelUpOverlay(PlayerState.getLevel());` and `Renderer.updateHud(PlayerState.getXp(), PlayerState.getLevel());` (the v1 HUD still shows XP/level even in dungeon mode)
    - If result.killed: call `App.transition('loot');` and return (no question advance)
    - If result.died: call `App.transition('dead');` and return (no question advance)
    - Otherwise (combat continues): `setTimeout(() => { CombatInputHandler.locked = false; CombatInputHandler.nextQuestion(); }, CONFIG.ADVANCE_DELAY_MS);`

    Method `nextQuestion()`:
    - Calls `QuestionSelector.selectNext(PlayerState)` and stores result as `q`
    - Sets `CombatInputHandler.currentQuestion = q`
    - Calls `DungeonRenderer.renderCombatQuestion(q)`

    After the CombatInputHandler const definition, add `CombatInputHandler.setup();` in the Bootstrap section, after `InputHandler.setup();`. This ensures both handlers are registered on page load.

    - `parseInt(selectedValue, 10)` radix must be specified — established project security pattern
    - The mode guard `App.mode !== 'dungeon'` must be the FIRST check in the change listener — before the locked check
    - CombatInputHandler does NOT call `PlayerState.updateAccuracy()` — combat answers do not affect the v1 per-table accuracy tracking in Phase 4 (Phase 5 may extend this; for now the combat loop is pure HP/XP)
    - CombatInputHandler does NOT call `PersistenceStore.save()` — XP is saved by the v1 persistence triggers (visibilitychange/beforeunload) already installed; no new save calls needed
    - The v1 `InputHandler.handleAnswer()` remains completely unchanged — its `if (App.mode !== 'quiz') return;` guard prevents it from firing during combat

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User click (radio change) → CombatInputHandler.handleAnswer | User-supplied radio value flows into parseInt(,10) then compared to stored integer — no further propagation to DOM |
| CombatEngine.resolveAnswer result → DungeonRenderer | Trusted internal module result; only numeric values (HP) and boolean flags cross this boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | Tampering | Radio input value parsing in handleAnswer | mitigate | `parseInt(selectedValue, 10)` with radix — established pattern from v1 InputHandler; comparison is integer equality, not string equality |
| T-04-03-02 | Tampering | Cross-handler radio group interference | mitigate | Combat inputs use `name="combat-answer"` (set in renderCombatQuestion); v1 uses `name="answer"` — distinct groups prevent cross-firing. Mode guard is first check in each handler. |
| T-04-03-03 | Elevation of Privilege | Combat loop awarding XP via v1 InputHandler.handleAnswer | mitigate | v1 InputHandler.handleAnswer has `if (App.mode !== 'quiz') return;` guard (verified present in Phase 3). CombatInputHandler does NOT call PlayerState.addXp() directly — XP via CombatEngine.resolveAnswer is the only path. |
| T-04-03-04 | Denial of Service | Locked state never releasing if killed/died path fires | mitigate | On killed/died, App.transition() is called and return immediately — no setTimeout is created, so lock does not need to release (new combat session via startCombat() sets fresh state). |
| T-04-03-SC | Tampering | npm/pip/cargo installs | accept | No package installs — pure inline JS edit. |
