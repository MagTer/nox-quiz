<objective>
Give the validator a coin-appropriate reachability model and prove it RED-first. The validator today checks biome, over-hole, spawn-goal, gap-width, mechanic-reachability and secret-alcove-reachability — it has NO coin check at all, which is exactly why unreachable coins passed every gate for four milestones.

The crux: a coin is NOT an alcove. `build.js` emits `add([sprite("coin"), pos(c.x, c.y), area(), "coin"])` — a 32x32 hitbox — and it is collected MID-ARC, in flight: the player only has to PASS THROUGH its box. The alcove model asks "can I get to and STAND at this zero-width point", which is a landing question. Probing coins through the alcove model produced a 32-coin figure that OVER-REPORTS. That number is an upper bound, never the work list.

Purpose: this plan lands the check and lets it go genuinely RED on the real offenders, re-deriving the true work list from a coin-shaped model. Plans 34-03/34-04/34-05 then move coins until it is GREEN.
Output: an extended `scripts/lib/reachability.mjs` (never a rewrite), a one-defect RED-first fixture, and the true unreachable-coin work list printed and recorded.
</objective>

For every node `n` already in `spawnPaths` (spawn-reachable — never re-derive that set), with `surfaceTop = n.y - PLAYER_H` (the player's top-left y while standing on n) and the takeoff range `x0min = n.xStart`, `x0max = Math.max(n.xStart, n.xEnd - PLAYER_W)`, evaluate exactly three physically-motivated witness families and return the best (preference order below) or `null`:

    - Family `walk` (t = 0): the player is simply standing/walking on n. Feasible iff `surfaceTop` lies inside the box's y-range AND `[x0min, x0max]` intersects the box's x-range. Witness `takeoffX` = the midpoint of that x-intersection, `dir` = +1, `t` = 0.
    - Family `jump` (t > 0): the player takes off from a free `x0` in `[x0min, x0max]` in direction `dir` (+1 and -1 both evaluated). Vertical position follows this module's existing convention (Kaplay y grows downward, so a rise is negative): `py(t) = surfaceTop + 0.5*CONFIG.GRAVITY*t**2 - CONFIG.JUMP_FORCE*t`. Horizontal: `px(t) = x0 + dir * envelope.runSpeed * t`.

      **PLAN-CHECK FIX (FLAG-1) — CLAMP THE RISE TO THE EMPIRICAL ENVELOPE.** Before sampling, reject the coin outright if it sits higher than the calibrated ceiling: `if (surfaceTop - box.y1 > envelope.maxRise) continue;` (i.e. apply the SAME `dy < -envelope.maxRise` cutoff that `jumpReach()` already applies at reachability.mjs:113), and additionally discard any sampled `t` whose `py(t)` rises more than `envelope.maxRise` above `surfaceTop`.

      Why this is not pedantry: raw `py(t)` has a theoretical apex of `JUMP_FORCE**2 / (2*GRAVITY)` = **96.57px**, but `JUMP_ENVELOPE.maxRise` is **88.331px** — and that number is `minObservedRise (92.98) * 0.95`, an EMPIRICALLY MEASURED ceiling from `calibrate-jump-envelope.mjs`, not a theoretical one. **The theoretical apex was never actually observed.** So an unclamped model does not merely run "generous" — it is WRONG: it would credit coins in the 88.3–96.6px band that the player cannot actually reach, contradicting `jumpReach()` and every authoring rule in `docs/LEVEL-DESIGN.md`.

And a pickup that demands an apex-perfect jump is precisely the kind of precision pressure this project forbids (no timers, no pressure, forgiving). Under-credit here; never over-credit. Sample `t` from `SAMPLE_DT` to `T_MAX` in `SAMPLE_DT` steps (`SAMPLE_DT = 1/240`, `T_MAX = 2.0` — module-level named constants, no magic numbers inline). Because `py(t)` is independent of `x0`, filter on y FIRST (cheap): skip any t whose `py(t)` falls outside the box's y-range. For each surviving t, solve the x-interval analytically — the player passes through the box iff `[box.x0 - dir*envelope.runSpeed*t, box.x1 - dir*envelope.runSpeed*t]` intersects `[x0min, x0max]`. Witness `takeoffX` = the midpoint of that intersection.
    - Family `fall` (t > 0, walk-off-the-ledge): identical to `jump` but with jump force 0 (`py(t) = surfaceTop + 0.5*CONFIG.GRAVITY*t**2`) AND with `x0` PINNED to the departure edge rather than free — `dir` +1 pins `x0 = x0max`, `dir` -1 pins `x0 = x0min` — because a fall only begins once the player actually walks off that edge. (A free `x0` here would fabricate a player falling out of the middle of a solid platform.)

    Preference order for the returned witness (simplest-to-replay first, so Plan 34-02's replay is robust): `walk` before `fall` before `jump`; within a family, the smallest `t`; ties broken by `dir` +1 before -1. Return `{ launchNodeId, launchY: n.y, takeoffX, dir, family, t }`.

The model therefore under-credits by ~8px of rise BY DESIGN, matching `jumpReach()` and `docs/LEVEL-DESIGN.md`. Note the asymmetry that makes this the most dangerous of the three: limitations 1 and 2 both under-credit (safe — they can only cause a coin to be moved that didn't strictly need moving), and limitation 1 is *caught* by 34-02's in-engine replay. An UNCLAMPED rise would be the only way this model OVER-credits, and 34-02 could not catch it — a scripted perfect hold-jump reaches the theoretical apex even though a child does not.

    **The invariant, stated once: this model must only ever UNDER-credit. If a coin is doubtful, it is unreachable.** The cost of under-crediting is a coin needlessly nudged 8px; the cost of over-crediting is a coin she can see and never get.
