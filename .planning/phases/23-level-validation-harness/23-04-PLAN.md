    Export `buildNodes(geometry)`: one node `{id: 'floor-'+i, xStart: f.x, xEnd: f.x+f.w, y: CONFIG.FLOOR_Y}` per entry in `geometry.floors ?? []`, one node `{id: 'platform-'+i, xStart: p.x, xEnd: p.x+p.w, y: p.y}` per entry in `geometry.platforms ?? []`.

    Export `nodeContaining(nodes, x, y)`: returns the first node whose `[xStart, xEnd]` contains `x`, optionally requiring `Math.abs(y - node.y) < 8` when `y` is supplied (disambiguates an overlapping floor/platform pair).

    Export `jumpReach(dy, envelope)`: if `dy < -envelope.maxRise`, return `[]` (physically cannot rise this high — the empirically-calibrated cutoff, never a locally re-derived closed-form ceiling). Otherwise compute the discriminant `disc = CONFIG.JUMP_FORCE**2 + 2*CONFIG.GRAVITY*dy` (the underlying projectile-motion algebra is still real physics from CONFIG, which is correct — only the maxRise CUTOFF above and the `runSpeed` multiplier below must come from `envelope`, not from a re-derived closed-form ceiling); if `disc < 0` return `[]`; compute both roots `t = (CONFIG.JUMP_FORCE ± sqrt(disc)) / CONFIG.GRAVITY`, keep only `t > 0`, and for each surviving root return `{t, reach: envelope.runSpeed * t}`. Additionally compute the same two roots with `CONFIG.JUMP_FORCE` replaced by `0` (the "step off the ledge, no jump needed" case per Pattern 1's "Falling-only edges" note) and union any of ITS positive-root reaches into the returned array too, so a pure walk-off-the-edge drop is covered by the same function without branching on the sign of `dy`.

    Export `canReach(fromNode, toNode, envelope)`: compute `dy = toNode.y - fromNode.y`; call `jumpReach(dy, envelope)`; for each candidate `{reach}`, check whether `toNode`'s span (`[toNode.xStart, toNode.xEnd]`, measured relative to `fromNode`'s near edge in the direction of travel — handle both left-of and right-of cases) contains that `reach` distance; if any candidate lands inside the span, return `{marginRatio: reach / (envelope.runSpeed * theoreticalMaxTAtThisDy)}` for the BEST (lowest-margin) matching candidate, where `theoreticalMaxTAtThisDy` is the larger of the two real roots computed above (the reach the player would get if they used the FULL calibrated envelope at this Δy); if no candidate matches, return `null`.

    Export `buildGraph(nodes, envelope)`: a `Map<nodeId, Array<{to: nodeId, marginRatio}>>` built by testing `canReach` between every ordered pair of distinct nodes (both directions — the player can jump either way).

    Export `bfsReachableSet(graph, startNodeId)`: a standard unweighted BFS/DFS over `graph`, returning the `Set` of node ids reachable from `startNodeId` (pure connectivity, no weights — every edge is boolean-feasible per Pattern 2's reasoning).

    Document in the module's header comment, as a plain design statement (not quoting any banned literal): this graph never models a door/mathGate/enemy barrier as a blocking node or edge, because this game's math mechanics have no lockout state and are always eventually passable once the player physically reaches their floor run.

Exercise at minimum the four documented behavior cases above using small synthetic node/geometry fixtures constructed inline (do not import any real level).
