<task type="auto">
  <name>Task 1: dt-correctness audit of all hand-wired motion</name>
  <files>src/player.js, src/camera.js, src/scenes/game.js, src/config.js</files>
  <read_first>
    - .planning/phases/08-platformer-core-movement-physics-camera/08-RESEARCH.md (Pattern 6 frame-rate independence, Pitfall 4 frame-rate-dependent feel — the exact double-scale / raw-lerp anti-patterns)
    - src/player.js, src/camera.js, src/scenes/game.js (audit targets)
  </read_first>
  <action>
    Audit every per-frame motion path against the RESEARCH Pattern 6 rules and fix any
    violation found (fixes go to the offending module; any tunable value goes to CONFIG):
    (1) Horizontal run sets player.vel.x from CONFIG.RUN_SPEED with NO dt() multiplier —
    body() already integrates vel with dt() in move(); a dt() multiplier here is a
    double-scale bug. (2) Gravity is engine-integrated via setGravity/body() — confirm there
    is NO hand-rolled `vel.y += g * dt()` anywhere (that loses the maxVelocity clamp and
    physics events). (3) The coyote and buffer timers decrement by dt() (seconds) — confirm
    they are not decremented by a raw per-frame constant. (4) The camera uses
    `1 - Math.exp(-CONFIG.CAM_RATE * dt())` — confirm there is NO raw constant-factor lerp
    (e.g. lerp(a, b, 0.1)) anywhere, which would converge faster at higher refresh and break
    MOVE-05. If any violation exists, correct it; if all are clean, record "audit clean — no
    changes" in the SUMMARY. Do not introduce inline literals — any value that needs tuning
    moves to CONFIG.
  </action>
  <verify>
    <automated>node --check src/player.js && node --check src/camera.js && node --check src/scenes/game.js && grep -q "Math.exp(-CONFIG.CAM_RATE" src/camera.js && ! grep -Eq "vel\.x[ ]*=[^;]*\bdt\(\)" src/player.js && ! grep -Eq "vel\.y[ ]*\+=" src/player.js src/scenes/game.js && ! grep -Eq "lerp\([^,]+,[^,]+,[ ]*0\.[0-9]+\)" src/camera.js src/player.js src/scenes/game.js && echo OK</automated>
  </verify>
  <acceptance_criteria>
    - All three motion modules pass `node --check`
    - No dt() multiplier on vel.x assignment (no double-scale)
    - No hand-rolled `vel.y +=` gravity anywhere
    - No raw constant-factor lerp in any motion module; camera uses the exp(-rate*dt) form
    - Any tune change is CONFIG-only
  </acceptance_criteria>
  <done>Every hand-wired motion path is confirmed dt-correct (or fixed to be); no double-scaling, no raw-constant lerp, no hand-rolled gravity; tunables stay in CONFIG.</done>
</task>

    1. Serve the game: from the project root, `cd src && python3 -m http.server 8000`, then

    3. Now throttle the frame rate to simulate a non-60Hz display. Use ONE of:

       floatier/heavier). Jumps must not be eaten and the camera must not speed up.

    6. If anything differs across refresh rates, that is a dt-correctness bug — report exactly
       which behavior changed (run / jump / camera) so it can be fixed in CONFIG or the
       relevant module before re-verifying.
