<decisions>
## Implementation Decisions

### Sequencing — prototype pair 1-2 first, THEN 3-8 (user decision)
- **Rebuild levels 1 + 2 FIRST** (level-01 = calm swamp intro / soft landing; level-02 = first
  "intense + vertical" even level + the first key-as-math-skip reward), get a **human-verify
  sign-off**, and ONLY THEN rebuild levels 3-8 to the proven shape. This is deliberate rework
  insurance: it discards kid-validated content, so Phase 38's kid-UAT must budget for rejection —
  proving the shape on one calm + one intense archetype before committing 6 more levels de-risks
  that.
- **Consequence for planning:** 34.6 has a REAL mid-phase human-verify checkpoint after levels
  1-2. Do NOT rubber-stamp it (standing precedent). Consider whether to SPLIT 34.6 into
  34.6 (mechanic change + levels 1-2 + checkpoint) and a follow-on phase for 3-8 — the planner
  should propose the cleanest structure; a mid-phase `checkpoint:human-verify` gate is acceptable.
- The sign-off ideally involves the kid (she is the user), but at minimum the user; this is the
  first look at fully-new geometry.

### Level shape — all levels DISTINCT; even = intense + vertical (user clarification)
- **Every level must be DIFFERENT from every other level** — not just 07/08. The user flagged the
  LVL-02 requirement ("07 and 08 must differ from each other") as too narrow / a "bug in the
  requirements": ALL levels should be distinct.
- **Even-numbered levels (2, 4, 6, 8) are MORE INTENSE and VERTICAL; odd levels (1,3,5,7) are the
  calmer biome intro.** This generalizes the biome-pair rhythm into a firm rule. Biome pairs stay:
  1-2 swamp, 3-4 town, 5-6 cemetery, 7-8 castle.
- The gentle difficulty ramp 1→8 is preserved (level-01 stays a soft landing for a 12-year-old);
  calm→harsh is a design property, authored intentionally.
- Also agreed (level-shape brief, STATE.md 2026-07-15): **descents** (levels may go DOWN, not only
  up), **optional high routes** (risk/reward — the key lives on one), and **VISIBLE backtracking
  only** (no hidden routes; she must always be able to SEE where she's going — ADHD-safe).

### The KEY mechanic, REFINED — "math-skip token" (user decision; changes 34.5's shipped design)
- **On even/key-lock levels, the key is a math-SKIP reward, NOT a physical lock.** This OVERRIDES
  Phase 34.5's shipped semantics (34.5 made the key open a solid physical lock guarding the exit,
  with the end math gate ALWAYS required, key = additive 4th barrier). The user's refinement:
  - The key is an **OPTIONAL collectible** placed on a harder / high (risk-reward) route.
  - **NO physical wall / lock barrier** on these levels.
  - At the level end: **key-held → level clears FREE (skip the end math question); no key → answer
    the end math question to finish** (the normal path).
  - **Skipping via the key gives FULL level-clear XP/credit** — finding the key IS the achievement,
    no penalty for not doing the final math. (Simplest, cleanest reward framing — user chose "full
    credit" over a bonus-XP variant.)
- **This is a real change to the game's thesis** ("multiplication is the gate to progress" → on
  even levels the FINAL gate is skippable by exploring). Accepted knowingly by the user as a
  reward-for-exploration, ADHD-friendly design. The math door + enemy mid-level still stand; only
  the END gate is skippable, and only on even levels, and only if the key was found.
- **Softlock risk: NONE from the key.** Because the key is purely optional (missing it → normal
  math path, never a dead end), there is no lock to be stuck behind. This is much safer than
  34.5's physical lock. The Phase-34.5 physical-lock code + its HARD `key-lock-check.mjs` softlock
  validator REMAIN in the codebase (dormant for math-skip levels — no `geometry.locks` placed
  here; still guard any future physical-lock use). Do NOT delete them.

### Mechanic change required (CODE, land it before/with the level authoring)
- Make the **end-of-level gate key-conditional**: reuse Phase-34.5's key pickup + closure-local
  `keyHeld`/`heldKeyIds` run-state + HUD indicator (all shipped); extend the end-gate wiring
  (`src/scenes/game.js` onReachGoal / `src/ui/mathGate.js`) so that if the key is held, the level
  clears immediately with full XP INSTEAD of opening the end math challenge; else the end math
  challenge runs as today.
- **Levels use `geometry.keys` WITHOUT `geometry.locks`** for this math-skip usage (the key is a
  free-standing collectible; no paired physical lock). Confirm the builder + validator tolerate a
  key with no lock (34.5's `?? []` guards should already allow it — verify, and if the softlock
  check assumes a paired lock, make a keys-without-locks level a clean PASS, not a fail).
- **Audit coverage (both paths):** a driven player who COLLECTS the key finishes WITHOUT the math
  challenge (new path); a driven player who does NOT collect the key gets the end math gate. The
  static model is not sufficient (milestone precedent).

### Math density
- Still **1 door + 1 enemy + end gate = 3 challenges/level** (LOCKED). On EVEN levels the end
  challenge is skippable via the key reward (math OR key). Odd levels unchanged (end is always
  math). Twice the platforming per math gate is the intent, not dilution.

### Verification bar (success criteria)
- Structural validator + coin-reachability + the bidirectional in-engine drive ALL green with
  **zero HARD-FAILs across all 8 rebuilt levels** — every level provably completable from spawn by
  a real driven player, every coin provably collectable. This is where the 13 currently-deferred
  Phase-34 HARD-FAILs (8 coin levels 01-03, 5 headroom level-07) finally get DELETED by the
  rebuild — they exist on geometry this phase replaces.
- Full LEVEL-DESIGN.md compliance authored from the start (gap/rise/overlap/spacing/checkpoint +
  the headroom rule `rise − thickness − 32 ≥ 24px`), NOT retrofitted.
- Each level's explicit `bounds.right` hand-bumped to the new extent (bounds-convention trap).

### Claude's Discretion
- Exact per-level geometry, where on each even level the key's high route sits, the precise
  descent/high-route shapes, and the end-gate-key-conditional code structure — all at Claude's
  discretion within the rules above, surfaced to the user at the 1-2 prototype checkpoint.
