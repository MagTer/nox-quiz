<objective>
Close VERIFICATION.md's secondary VERIFY-02 gap: New Finding 4 (simultaneous/overlapping
challenges) was only half-fixed by Plan 21-04. The state-corruption half (a resolved
challenge's `destroyAll()` wiping out a different, still-open challenge) was fixed via
per-instance `instanceTag` scoping. The VISUAL-overlap half — two challenge overlays rendering
on top of each other, arguably the closer match for the original live-reported symptom "not
possible to answer... no ID on the options" — remains open, marked in `challenge.js`'s own
inline comment as "still out of scope here." Add a same-time-open visual guard to
`openChallenge()` that hides (not destroys) an already-open challenge's objects while a new one
is on top, and restores them when the new one closes.

Purpose: this is a real, ordinary-play-reachable bug (walking from an unresolved collect-the-
answer zone into a second mechanic before resolving the first) that this phase's own interactive
audit discovered. VERIFICATION.md explicitly allows either fixing it now (if tractable) or an
explicit, documented deferral — a naive fix is NOT tractable without risking a worse regression
(see the reasoning in `<truths>` above), so this plan implements the specific hide/restore design
that avoids that regression, confirmed via the vendored Kaplay engine source: every GameObj has a
base `hidden` boolean (`draw(){ if (this.hidden) return; ... }`) independent of any component, so
setting it on objects that never had an explicit `opacity()` component still reliably hides them.

Output: a fixed `src/ui/challenge.js`, an updated `21-FINDINGS.md` disposition, and screenshot
evidence proving the overlap no longer renders.
</objective>

<interfaces>
Kaplay base GameObj contract (confirmed via lib/kaplay.mjs — every entity, regardless of which
components it was created with, carries these two base properties):
  obj.hidden   — boolean, default false; draw() unconditionally returns early when true, so the
                 object stops rendering but is NOT destroyed, keeps updating, and can be
                 un-hidden later by setting it back to false. Safe to use even on objects that
                 were never given an explicit opacity() component (challenge.js's dim/prompt/
                 panel/box objects fall in this category today).
  get("challenge")  — returns every currently-live object carrying the "challenge" tag, across
                 ALL open instances (the generic tag every instance's objects share, per
                 src/ui/challenge.js's own existing design; instanceTag is the PER-INSTANCE tag
                 used for that instance's own destroyAll() teardown).

Only ONE non-freezing challenge (collect.js's zone) can ever be open at the same time as a
SECOND, freezing challenge (door/gates/enemy all set player.paused = true immediately on
trigger, which halts onCollide entirely — so a frozen player cannot physically trigger a THIRD
mechanic). Maximum concurrent-open depth is therefore always exactly 2, never more.
</interfaces>
