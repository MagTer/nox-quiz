<interfaces>
Physics constants this plan's traversal design is derived from (src/config.js, unchanged by
this plan): RUN_SPEED 240 px/s, GRAVITY 1400 px/s^2, JUMP_FORCE 520 px/s. A single jump's
time-to-apex is JUMP_FORCE/GRAVITY ~= 0.371s; max single-jump RISE is JUMP_FORCE^2/(2*GRAVITY)
~= 96.6px; full up+down hangtime ~= 0.743s; max single-jump horizontal travel at RUN_SPEED
~= 178px. Every authored compound gap in this game (confirmed against src/levels/level-0{1..4}.js
geometry) is crossable as a CHAIN of individual hops each within this ~96.6px rise / ~178px
horizontal envelope — e.g. level-01's gap needs floor(y:320) -> platform(y:240, rise 80px) ->
platform(y:192, rise 48px) -> floor(y:320, a fall) — never a single ~128px+ rise in one jump.

src/player.js's jump is a single JUMP_FORCE impulse fired from `player.onUpdate()` only when
`player.isGrounded() || coyote > 0` AND a buffered press exists (`buffer` set by
`onKeyPress(JUMP_KEYS, ...)`). Horizontal `vel.x` is set to `dir * CONFIG.RUN_SPEED` every
frame purely from held-key state, decoupled from jump/airborne state — so holding ArrowRight
always advances x at a constant rate whether grounded or airborne, and pressing Space is safe
and harmless at any time (it only queues a jump that fires on the next grounded/coyote frame,
per BUFFER_MS/COYOTE_MS). This is why "jump whenever grounded, else wait" is a safe, general
substitute for a precomputed gap model: a real player holding right and mashing jump would climb
the same compound sequences this way.

Kaplay tags/state to query via page.evaluate(() => ...) inside the browser page context:
  get("player")[0].pos.x       — live player x
  get("player")[0].isGrounded() — live grounded state (same method src/player.js itself calls)
  get("challenge").length      — count of all currently-open challenge-tagged objects (any
                                  instance); compare against a captured baseline to detect a
                                  NEW trigger (a stale, deliberately-left-open collect-zone
                                  challenge must not read as a false "already triggered")
</interfaces>
