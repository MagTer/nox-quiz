### Pattern 1: Key pickup — mirror `secretAlcove.js`

**What:** A walk-through `area()` trigger that fires once, gives feedback, destroys itself, and sets scene run-state. NO pause, NO challenge, NO XP.

`src/mechanics/secretAlcove.js` (read in full) is the template. The KEY half is that file with three changes:
1. Replace `progress.addBonusXp(...)` / `progress.markSecretFound(...)` / the `hasSecretFound` replay-guard with a single call to an injected `onPickup()` callback that flips the scene's `keyHeld` flag.
2. Replace the `+5 XP` popup text with a `"KEY"` (or `"+ KEY"`) popup.
3. Drop the cross-run persistence entirely — key-held is *run* state, never serialized (matches `coinsCollected`/`goalReached` in `game.js`, which are never part of the save).

Reuse verbatim: the closure-local `const collected = new Set()` fire-once latch (`secretAlcove.js:47`), `fx.pop(obj.pos.clone())` + `fx.popupText(...)` + `audio.playSfx("pickup")` feedback, `destroy(keyObj)` last.

```javascript
// src/mechanics/key.js — the KEY half (engine globals INSIDE the body only — a727c13)
import { CONFIG } from "../config.js";
import * as fx from "../fx.js";
import * as audio from "../audio.js";

export function wireKey({ player, hud, onPickup, hasKey }) {
  const collected = new Set();               // closure-local fire-once (secretAlcove.js:47)

  player.onCollide("key", (keyObj) => {
    if (collected.has(keyObj)) return;
    collected.add(keyObj);
    onPickup();                               // flips scene keyHeld = true
    hud.showKey();                            // persistent HUD indicator (Finding 4)
    fx.pop(keyObj.pos.clone());
    fx.popupText(keyObj.pos.clone(), "+ KEY");
    audio.playSfx("pickup");                  // reuse the existing pickup sfx (no new asset)
    destroy(keyObj);
  });
  // … the LOCK half below (Pattern 2) lives in this SAME wireKey() body …
}
```

### Pattern 2: Lock — mirror `door.js` MINUS the challenge

**What:** A genuinely-solid apex-height wall the player physically cannot pass until key-held; on collide it opens+destroys iff key-held, else shows a non-blocking hint.

`src/mechanics/door.js` (read in full) is the template. The critical differences from the door:
- **No `openChallenge()`, no `player.paused`, no velocity zeroing, no `busy` re-entrancy guard.** The door freezes the player to run a challenge; the lock never freezes — it is just a wall. So the whole freeze/unfreeze machinery is dropped.
- **The door.js "destroy BEFORE unfreeze" ordering still applies in spirit:** on a key-held collide, `destroy` the blocker + its panel immediately so the same-frame overlap cannot re-block. There is no unfreeze, but destroying the collider is what lets the player continue through the now-open doorway that same touch session.
- **The hint must be fire-once-ish without a timer.** onCollide fires every overlap frame; guard the hint with a closure-local boolean cleared by the hint's own `tween().onEnd()` (the no-timer self-clean idiom — `hud.js:135` / `game.js:255` precedent). Do NOT use `setTimeout`/`wait()`/`loop()` (banned, `check-safety.sh`).

```javascript
// src/mechanics/key.js — the LOCK half, inside the same wireKey() body
const opened = new Set();
let hintShowing = false;

player.onCollide("lock", (lockObj) => {
  if (opened.has(lockObj)) return;
  if (hasKey()) {
    opened.add(lockObj);
    audio.playSfx("door");                    // reuse door sfx (no new asset)
    fx.clearBurst();                          // optional celebratory beat (door.js precedent)
    // Destroy collider + panel BEFORE the next frame so the same touch can't re-block.
    destroy(lockObj);
    if (lockObj.panelObj) destroy(lockObj.panelObj);
    if (lockObj.glyphObj) destroy(lockObj.glyphObj);
    return;
  }
  // No key: solid wall stays. Show a brief, non-blocking hint (NO pause, NO challenge).
  if (hintShowing) return;
  hintShowing = true;
  const hint = add([
    text(CONFIG.LOCK.HINT_TEXT, { size: CONFIG.LOCK.HINT_SIZE }),
    anchor("center"),
    pos(center().x, CONFIG.LOCK.HINT_Y),
    color(CONFIG.PALETTE.TEXT[0], CONFIG.PALETTE.TEXT[1], CONFIG.PALETTE.TEXT[2]),
    fixed(), z(9200), opacity(1), "hud-flash",  // "hud-flash" → torn down on scene leave (anti-leak)
  ]);
  tween(1, 0, CONFIG.LOCK.HINT_MS / 1000, (v) => (hint.opacity = v), easings.easeOutQuad)
    .onEnd(() => { destroy(hint); hintShowing = false; });
});
```

### Pattern 3: build.js entity creation

Follow the two shipped emission idioms:
- **Key** = the `secretAlcove` walk-through idiom (`build.js:364-372`) but VISIBLE (the alcove is a secret; the key is telegraphed). A bare `area()` trigger + a visible colored rect (or a `sprite("key")` if art is added). Under `?debug=1`, render a bright marker like the alcove does.
- **Lock** = the door blocker idiom (`build.js:266-289`): a tall invisible apex-height solid collider `rect(CONFIG.LOCK.W, blockerH)` with `body({isStatic:true})`, `area()`, `opacity(HIDDEN)`, tag `"lock"`, plus a separate visible panel (`sprite` or colored rect) tagged `"lock-panel"` stashed as `blocker.panelObj`. Use the exact apex formula the door/gate/enemy blockers use:
  `const blockerH = Math.ceil((CONFIG.JUMP_FORCE ** 2) / (2 * CONFIG.GRAVITY)) + 64;` — this is what makes the lock genuinely unjumpable (build.js:273), which the validator's x-partition model depends on.

Both loops must be `?? []`-guarded (`for (const k of g.keys ?? [])`, `for (const l of g.locks ?? [])`) so every existing level still builds — the never-brick convention.

### Anti-Patterns to Avoid
- **Do NOT make the lock open a challenge or pause the player.** It is a spatial gate, not a math gate (CONTEXT + math-density-3 lock).
- **Do NOT revive `collect.js`.** `collect.js` was deleted in Phase 29 as a bad fit; the key is a spatial traversal gate wired like the alcove, NOT a collect-the-answer mechanic (CONTEXT `<specifics>`).
- **Do NOT put `keyHeld` at module level.** Anti-leak: closure-local in `game.js`, GC'd with the scene, and it survives `respawn()` (reposition-in-place, no `go()`) for free.
- **Do NOT model the lock as a small gap in the validator.** A gap is jumpable; the lock reaches jump apex. Model it as an impassable x-partition (Finding 1).
- **Do NOT add a `setTimeout`/`wait()`/`loop()` for the hint.** Self-clean via `tween().onEnd()`.

## Common Pitfalls

### Pitfall 1: The lock is co-located with the goal / fires too late
**What goes wrong:** the end math-gate `go("select")`s on a correct answer (`game.js:258`), tearing down the scene. If the lock sits AT the goal it is never evaluated.
**How to avoid:** place the lock guarding the corridor BEFORE the goal (CONTEXT ordering: locked corridor → math gate → clear). In level-02 the goal is at x:4200 on floor run `{x:3680,w:600}` (3680..4280); place the lock earlier on that same run (e.g. x~3960) so the player must open it to reach x:4200.

### Pitfall 2: `check-import-safety.sh` does not scan `key.js` — the a727c13 trap goes unguarded
**What goes wrong:** the import-safety gate has a HARDCODED file list (`check-import-safety.sh:84-89`) covering door/gates/enemy/secretAlcove but NOT a future `key.js` (nor `hud.js`). A top-level engine-global reference in `key.js` would blank the game and no gate would catch it.
**How to avoid:** add `"src/mechanics/key.js"` to that `for f in \ … ` list as part of this phase. Keep every engine global (`add`, `text`, `destroy`, `tween`, `center`, `pos`, …) inside the `wireKey()` body.

### Pitfall 3: Hint spams every overlap frame / uses a banned timer
**What goes wrong:** `onCollide` fires continuously while the player leans on the wall; a naïve hint stacks dozens of text objects, and reaching for `setTimeout`/`wait()` to dismiss it trips `check-safety.sh`.
**How to avoid:** the closure-local `hintShowing` boolean gate + `tween().onEnd()` self-clean shown in Pattern 2. Tag the hint `"hud-flash"` so a scene leave sweeps it (anti-leak, like `flashLevelUp`).

### Pitfall 4: New sprite path without a manifest entry hard-fails the asset gate
**What goes wrong:** `node scripts/check-assets-manifest.mjs` fails if a declared sprite path isn't in `src/assets-manifest.js`, and Kaplay throws on an unloaded `sprite("key")`.
**How to avoid:** use **colored-rect placeholders** for key + lock panel (art is deferred to Phase 35 — CONTEXT permits placeholder visuals). This adds ZERO assets and ZERO manifest/`main.js` changes. If a sprite is genuinely wanted, it must be added to BOTH `src/assets-manifest.js` AND a `loadSprite` in `src/main.js`. Reuse the existing `pickup`/`door` SFX — no new sounds either.

### Pitfall 5: Static-model-only "proof" ships a bug
**What goes wrong:** this milestone has repeatedly shipped bugs past a green static model (sawtooth floor, grey ground, ceiling-bonk coins). A green `key-lock-check.mjs` alone is not proof (SC3).
**How to avoid:** the `audit-key-lock.mjs` driven-player gate is a REQUIRED deliverable, not optional — assert a real driven player collects the key then opens the lock (`triggered:true`).

## Code Examples

### Finding 1: The softlock validator — `checkKeyLockReachability(geometry)`

The existing reachability graph **deliberately never models barriers as blocking** (reachability.mjs header, lines 22-29): doors/gates/enemies have no lockout state, so the full graph already reaches the goal *ignoring* the lock — which is correct for the "with the key, the lock is open" far-side case. The NEW check needs a SEPARATE, lock-cut graph to prove the key is on the spawn side.

**Algorithm (recommended, reuses existing helpers verbatim):**

```javascript
// scripts/lib/key-lock-check.mjs  (PURE — no engine globals, node-importable; a727c13-safe)
import { fileURLToPath } from "url";
import { CONFIG } from "../../src/config.js";
import { JUMP_ENVELOPE } from "./jump-envelope.mjs";
import {
  buildNodes, buildGraph, bfsWithPathMargin, bestMarginToPoint,
  nodeContaining, SPAWN_X, WARN_MARGIN_RATIO,
} from "./reachability.mjs";
// NOTE: bfsWithPathMargin is currently module-private in reachability.mjs (line 323).
// Export it there (one-line change) OR inline an equivalent BFS here. Prefer exporting —
// bestMarginToPoint already needs a spawnPaths map of exactly its shape.

const LOCK_W = CONFIG.LOCK.W; // apex-height blocker footprint width (see build.js/config)

export function checkKeyLockReachability(geometry, envelope = JUMP_ENVELOPE) {
  const rows = [];
  const nodes = buildNodes(geometry);
  const locks = geometry.locks ?? [];
  const keys  = geometry.keys  ?? [];

  for (const [i, lock] of locks.entries()) {
    // Match this lock to its key: same keyId if present, else the sole pair (index 0).
    const key = lock.keyId != null
      ? keys.find((k) => k.keyId === lock.keyId)
      : keys[i] ?? keys[0];
    if (!key) {                       // a lock with no key is an unconditional softlock
      rows.push({ check: "key-lock", status: "HARD-FAIL",
        descriptor: `lock[${i}] x:${lock.x} has no matching key` });
      continue;
    }

    // --- Build a LOCK-CUT graph: drop every edge whose endpoints straddle the lock's
    // x-band [lock.x, lock.x + LOCK_W]. The lock blocker reaches jump apex (build.js:273),
    // so NO arc can cross it — an edge from a node entirely left of the band to a node
    // entirely right of it (or vice versa) is genuinely impassable while the lock stands.
    const bandLo = lock.x, bandHi = lock.x + LOCK_W;
    const fullGraph = buildGraph(nodes, envelope);
    const cutGraph = new Map();
    const byId = new Map(nodes.map((n) => [n.id, n]));
    for (const [fromId, edges] of fullGraph) {
      const from = byId.get(fromId);
      cutGraph.set(fromId, edges.filter(({ to }) => {
        const t = byId.get(to);
        const leftThenRight  = from.xEnd <= bandLo && t.xStart >= bandHi;
        const rightThenLeft  = t.xEnd  <= bandLo && from.xStart >= bandHi;
        return !(leftThenRight || rightThenLeft);   // keep every non-crossing edge
      }));
    }

    // --- Spawn-side reachable set over the CUT graph, then the alcove point-reach model.
    const spawnNode = nodeContaining(nodes, SPAWN_X);
    const spawnPaths = spawnNode ? bfsWithPathMargin(cutGraph, spawnNode.id) : new Map();
    const reach = bestMarginToPoint({ x: key.x, y: key.y }, nodes, spawnPaths, envelope);

    if (reach === null) {
      // Key not reachable from spawn WITHOUT crossing the lock → the worst-case softlock.
      rows.push({ check: "key-lock", status: "HARD-FAIL",
        descriptor: `key[${i}] x:${key.x} y:${key.y} NOT reachable before lock x:${lock.x} (softlock)` });
    } else {
      rows.push({ check: "key-lock",
        status: reach.marginRatio >= WARN_MARGIN_RATIO ? "WARN" : "PASS",
        descriptor: `key[${i}] x:${key.x} reachable spawn-side of lock x:${lock.x} (marginRatio=${reach.marginRatio.toFixed(3)})` });
    }
  }
  const hardFailCount = rows.filter((r) => r.status === "HARD-FAIL").length;
  return { rows, hardFailCount };
}
// + a self-test (isMain) block mirroring over-hole-check.mjs:71-127 / reachability.mjs:1170+.
```

**Graph model answers (from reading reachability.mjs in full):**
- **Nodes** = one per floor run + one per platform, `{ id:"floor-N"|"platform-N", xStart, xEnd, y }` (`buildNodes`, lines 114-123). Floors sit at `CONFIG.FLOOR_Y` (320); platforms at their `p.y`.
- **Edges** = a directed edge A→B exists iff `canReach(A,B,envelope)` returns non-null — a real projectile-motion arc within the empirically-calibrated `JUMP_ENVELOPE` lands inside B's span (`canReach` lines 198-269; both directions tested by `buildGraph` lines 276-289, including leftward `toNode.xEnd <= fromNode.xStart`).

## Open Questions

1. **Should the far side / goal be provably UNreachable without the key (lock actually gates)?**
   - What we know: CONTEXT's HARD requirement is key-reachable-spawn-side (softlock safety). A lock that gates *nothing* (goal reachable by another route without the key) is not a softlock — it's just a pointless lock.
   - What's unclear: whether to add a WARN (not HARD) row when the goal is reachable in the lock-cut graph (i.e. the lock is bypassable/decorative).
   - Recommendation: add an optional WARN row "lock x:… does not gate the goal (bypassable)" — informative, never fails the suite. Not required for KEY-02.

2. **Multi-pair `keyId` matching self-test.**
   - What we know: the data model carries optional `keyId`; no level exercises it (deferred).
   - Recommendation: the `checkKeyLockReachability` `keyId` matching (shown in Finding 1) ships latent; add ONE self-test fixture case with two pairs so the matching logic is proven even though no real level uses it.
