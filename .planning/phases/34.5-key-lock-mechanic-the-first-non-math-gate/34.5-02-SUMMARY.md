# Phase 34.5 Plan 02: Key/Lock Mechanic Wiring Summary

**The key/lock mechanic exists as CODE (KEY-01): a new `src/mechanics/key.js` `wireKey()` seam wires both halves — a secretAlcove-style walk-through key pickup and a door.js-style apex-height solid lock that opens on a key-held collide (destroying collider + panel before the next frame) or shows a self-cleaning non-blocking hint otherwise — with `build.js` emitting the entities from `geometry.keys`/`geometry.locks`, `game.js` threading closure-local `keyHeld` run-state, and the HUD showing a persistent key-held indicator. All a727c13-clean, no timers, no punishment, no new assets. The mechanic is inert until a descriptor uses it (Plan 03).**

## Accomplishments

- **`src/mechanics/key.js`** (NEW) — `wireKey({ player, hud, onPickup, hasKey })` wires both halves in one body:
  - **KEY half** mirrors `secretAlcove.js`: a closure-local `collected` Set fire-once latch, `onPickup()` (flips scene `keyHeld`), `hud.showKey()`, the fx.pop / fx.popupText("+ KEY") / audio.playSfx("pickup") feedback triad (reusing the existing pickup SFX), `destroy(keyObj)` last. All persistence dropped — key-held is run-state, never serialized.
  - **LOCK half** mirrors `door.js` MINUS the freeze/challenge machinery (no `busy`, no `player.paused`/vel-zeroing, no `openChallenge`): an `opened` Set + a closure-local `hintShowing` boolean. On a key-held collide it plays the door SFX, clearBurst, and destroys the collider then `panelObj` BEFORE the next frame (door.js:74-79 ordering) so the same touch cannot re-block. On a keyless collide it shows a brief `fixed()` `"hud-flash"`-tagged hint that fades and self-destroys via `tween().onEnd()` — the ONLY dismissal path, gated by `hintShowing` so it cannot spam every overlap frame. NO damage / bounce / reset / game-over.
  - All engine globals (`add`, `text`, `destroy`, `tween`, `center`, `pos`, `color`, `anchor`, `opacity`, `z`, `fixed`, `easings`) live INSIDE the `wireKey` body only (a727c13).

## Decisions Made

- **Lock uses the exact door apex formula.** `Math.ceil((CONFIG.JUMP_FORCE ** 2) / (2 * CONFIG.GRAVITY)) + 64` — identical to the door/math-gate/enemy blockers — so the lock is genuinely unjumpable. This keeps Wave 1's `key-lock-check.mjs` x-band partition sound: a shorter lock would let a jump arc cross above the band and the validator's "spawn-side reachable" proof would be unsound.
- **Lock panel is a colored rect, not a sprite.** Reusing `sprite("door")` would be wrong art, and a new sprite would churn the asset manifest. `CONFIG.LOCK.PANEL_COLOR` (PALETTE.MUTED from Wave 1) is a placeholder; real art is Phase 35.
- **`keyHeld` threaded as callbacks, not by value.** `onPickup: () => { keyHeld = true; }` and `hasKey: () => keyHeld` mean the lock handler always reads the LIVE flag regardless of collision-event ordering, and the flag stays closure-local (never leaks, survives respawn, never serialized).
- **Reused SFX.** Key pickup reuses `pickup`, lock-open reuses `door` — no new sounds (matches the alcove/door precedent; distinct SFX is a Phase-35 polish concern).
