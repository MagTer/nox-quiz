# Phase 34.5 Plan 01: Key/Lock Softlock Validator Summary

**A sound, RED-first-proven static reachability check (`checkKeyLockReachability`) that HARD-proves every key/lock pair's key is reachable from spawn before its lock, via a per-lock lock-cut BFS graph that clips every node at the lock's x-band and drops cross-band edges — built and composed into `validate-levels.mjs` before any mechanic code or level content exists.**

## Accomplishments

- `src/config.js` gained `CONFIG.LOCK` (W/H/PANEL_COLOR/HINT_TEXT/HINT_SIZE/HINT_Y/HINT_MS), `CONFIG.KEY` (W/H/COLOR), and `CONFIG.HUD.KEY_X/KEY_Y/KEY_SIZE/KEY_GLYPH` — the shared tunables Wave 2's mechanic module and HUD will consume; `bfsWithPathMargin` and `bestMarginToPoint` are now exported from `reachability.mjs` (both were module-private).
- `scripts/lib/key-lock-check.mjs` is a new, pure, node-importable module exporting `checkKeyLockReachability(geometry)`. It fixes the naive-model bug the plan-checker flagged: `buildNodes` emits ONE node per floor run, so a lock mid-run never splits it under a plain edge-drop model. This check instead clips EVERY node's span (floor or platform) at the lock's x-band first, then drops any surviving edge whose endpoints land on opposite sides of the band — sound because the lock's in-engine collider is apex-tall (mirrors `build.js`'s door/mathGate/enemy blocker formula), so no jump arc can cross above it.
- A 5-case `isMain` self-test proves the fix fires: (a) same-floor-run key-behind-lock → exactly one HARD-FAIL (the regression guard for the naive-model bug), (b) legit spawn-side-reachable key → PASS/WARN, (c) a low platform bridging near the band → still HARD-FAILs (the A2 loophole guard), (d) two key-lock pairs disambiguated by `keyId` listed out of positional order → both correctly matched and reachable, (e) omitted `keys`/`locks` → never throws, returns the empty-clean contract.
- `scripts/fixtures/bad-level-key.js` is the RED-first proof: a single flat floor run with a key sealed behind its own lock. `node scripts/validate-levels.mjs --fixture scripts/fixtures/bad-level-key.js` exits 1 with exactly one `key-lock | HARD-FAIL` row.
- `scripts/validate-levels.mjs` now composes `checkKeyLockReachability` into its per-descriptor loop (after `checkLevelReachability`), summing `hardFailCount` into `failures` exactly like every other check. All 8 real levels print zero key-lock rows (none carry keys/locks yet) and show the same 13 pre-existing HARD-FAILs Phase 34 deliberately deferred to 34.6 — no regression.

## Decisions Made

- **Generalized node-clipping to ALL node types (floor + platform), not just floor nodes.** The plan's algorithm description said "split every floor node whose span strictly contains the band," but self-test case (c) (a platform bridging near/over the band) exposed that a platform-typed node needs the identical clip treatment — otherwise a low platform positioned to overlap or sit fully inside the lock's band could still offer an unguarded bypass. `clipNodesAtBand` in `key-lock-check.mjs` operates on every node in `buildNodes()`'s output regardless of `id` prefix, closing this gap uniformly.
- **A node fully embedded inside the band is dropped entirely** (zero remainder pieces), rather than kept as a zero-or-negative-width fragment — matching the physical reality that nothing embedded in the lock's own footprint is a usable standing surface.
- **LOCK.PANEL_COLOR reuses `PALETTE.MUTED`** (the existing DOOR/MATH_GATE locked-tile convention) rather than a new literal, per the plan's suggestion — keeps the placeholder visually consistent with other locked barriers pending Phase 35 art.
