    Edit the existing PersistenceStore IIFE in-place (do not create a new module, do not duplicate it). Make only these targeted changes:

    CHANGE 1 — VERSION constant: Change `const VERSION = 1;` to `const VERSION = 2;`

    CHANGE 2 — KEY constant: Change `const KEY = CONFIG.SAVE_KEY;` to `const KEY = CONFIG.DUNGEON.SAVE_KEY_V2;`
    (CONFIG.DUNGEON.SAVE_KEY_V2 is 'mathlab_save_v2', set by Plan 01 Task 1)

    CHANGE 3 — Add private migrate() function inside the IIFE, before the `return { ... }` statement:

    The migrate() function reads the v1 save, validates it, and writes the migrated data to the v2 key:

    - Read raw from `localStorage.getItem(CONFIG.SAVE_KEY)` (the v1 key: 'mathlab_save_v1')
    - If raw is null, return null (no v1 data to migrate)
    - Parse the raw JSON inside a try-catch; return null on parse failure
    - Validate: data.version must equal 1 (exact integer check); return null on mismatch
    - Validate: data.xp must be a non-negative finite number; data.level must be a positive integer
    - On validation failure, log `console.warn('[MathLab] v1 save invalid — skipping migration')` and return null
    - Build a v2 payload: `{ version: 2, xp: data.xp, level: data.level, accuracy: data.accuracy || {}, history: data.history || {} }`
    - Write v2 payload to localStorage: `localStorage.setItem(KEY, JSON.stringify(v2Payload))`
    - Log `console.info('[MathLab] Migrated save from v1 → v2')` on success
    - CRITICAL: Do NOT call localStorage.removeItem on the v1 key — it must remain untouched
    - Return v2Payload on success

    CHANGE 4 — Modify load() to call migrate() when v2 key is absent:

    Inside load(), after the existing `const raw = localStorage.getItem(KEY);` line, add a migration branch:

    ```
    if (raw === null) {
      // No v2 save found — attempt migration from v1
      const migrated = migrate();
      if (migrated !== null) {
        // Migration succeeded — load the freshly written v2 data
        PlayerState.fromJSON(migrated);
        return migrated;
      }
      return defaults();
    }
    ```

    The rest of the existing load() body (JSON.parse, version check, numeric field validation) remains unchanged except the `data.version !== VERSION` check now correctly rejects any non-2 version.

    CHANGE 5 — defaults() function: Add `version: VERSION` to the returned object to ensure defaults always carry the current version. The existing defaults() already has `version: VERSION` — verify it is present and unchanged.

    Do not modify save() — it already uses KEY and VERSION by reference; after the constant changes it will automatically write to mathlab_save_v2 with version: 2.

    Add a comment above the migrate() function: `// Auto-migration from v1 save — runs once per user; v1 key is preserved`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localStorage v1 data → migrate() | v1 save data is user-controlled (could be tampered in DevTools before migration) |
| migrate() → localStorage v2 write | Output of migration written back to localStorage without further sanitization beyond fromJSON |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Tampering | migrate() reads from mathlab_save_v1 | mitigate | Validate data.version === 1 (strict check), data.xp as non-negative finite number, data.level as positive integer before writing v2; on failure log warn and return null (falls back to defaults) |
| T-02-09 | Tampering | Prototype pollution via migrated accuracy/history objects | mitigate | PlayerState.fromJSON already validates each key with `parseInt(k, 10)` range check (table 1–9) and type check — migration feeds data through fromJSON, not via Object.assign spread onto prototype |
| T-02-10 | Denial of Service | migrate() called repeatedly if v2 write fails | mitigate | load() checks for v2 key first; migration branch runs only when v2 is absent; if localStorage.setItem throws QuotaExceededError in migrate(), wrap in try-catch and return null (no loop possible) |
| T-02-11 | Information Disclosure | v1 key left in localStorage | accept | v1 key contains XP/level data already present in v2; no new information exposure; required for rollback ability; local-only app with no server exfiltration path |
| T-02-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase |
</threat_model>
