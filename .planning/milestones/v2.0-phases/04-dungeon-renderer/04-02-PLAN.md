    Insert MODULE 11 DungeonRenderer as an IIFE between `window.App = App;` and the `document.addEventListener('DOMContentLoaded', ...)` call.

    The module is a const IIFE: `const DungeonRenderer = (() => { ... return { publicApi }; })();` followed immediately by `window.DungeonRenderer = DungeonRenderer;`

    Private closure state the module must declare:
    - `let _currentEnemyMaxHP = CONFIG.DUNGEON.GOBLIN_HP;` — tracks max HP for percentage calc; updated in renderCombat()
    - `const lastFlavorIndex = {};` — tracks last shown flavor line index per enemy name to prevent immediate repetition
    - FLAVOR object with three keys: `Goblin`, `Skeleton`, `Dragon` — each an array of exactly 3 placeholder strings. Each array must have a code comment reading `/* PLACEHOLDER — Phase 6 replaces with user-approved content */` to mark them for easy Phase 6 replacement. The placeholder text for each enemy must use RPG-framing and must NOT contain the words: correct, wrong, answer, question (case-insensitive check is performed in Phase 6).
      - Goblin lines: "The goblin snarls at you.", "It bares its rotten teeth.", "The goblin charges forward."
      - Skeleton lines: "Bones rattle in the dark.", "Its hollow eye sockets stare.", "The skeleton raises its blade."
      - Dragon lines: "Smoke curls from its nostrils.", "The dragon lets out a low rumble.", "Its scales shimmer like obsidian."

    Private helper function `getFlavorText(enemyName)`:
    - Reads `FLAVOR[enemyName]` array; if missing or empty returns empty string
    - Uses a do-while loop to pick a random index that is NOT equal to `lastFlavorIndex[enemyName]` (only enforces no-repeat when lines.length > 1)
    - Stores chosen index in `lastFlavorIndex[enemyName]` and returns the line text

    Public API methods (all use textContent, never innerHTML):

    `renderCombat(state)` — accepts `{ floor, room, enemy, emoji, enemyHP, playerHP }`:
    - Sets `#combat-room-indicator` textContent to "Floor " + state.floor + " · Room " + state.room + "/5"
    - Sets `#combat-enemy-sprite` textContent to state.emoji
    - Sets `_currentEnemyMaxHP` to the enemy's current HP total from CombatEngine.getState().floorDef.hp — call `CombatEngine.getState()` here to read floorDef.hp; this is safe because renderCombat() is only called after startCombat()
    - Calls `updateHP(state.enemyHP, state.playerHP)` to initialize bars
    - Clears `#combat-feedback` textContent to empty string
    - Sets a `#combat-flavor` element textContent to `getFlavorText(state.enemy)` — NOTE: add `<p id="combat-flavor"></p>` to the combat panel HTML if it does not already exist (Plan 01 may not have included this; if the element does not exist, add it via Edit to the combat panel HTML section). The flavor text sits between the enemy area and the feedback area.

    `renderCombatQuestion(questionObj)` — accepts the question object from QuestionSelector.selectNext():
    - Clears `#combat-options-list` by setting innerHTML to empty string (innerHTML = '' for clearing the list container is the established pattern from Renderer.showQuestion — this is the ONLY innerHTML use allowed, and only for clearing, not for writing content)
    - Sets `#combat-feedback` textContent to empty string (implicit clear for previous feedback)
    - Sets combat flavor text: calls `getFlavorText` with the current enemy name (stored in a closure variable `_currentEnemy` set by renderCombat()) and writes to `#combat-flavor` textContent
    - Builds option items using createElement/appendChild — same pattern as Renderer.showQuestion() but targeting `#combat-options-list` with `name="combat-answer"` (NOT `name="answer"`) and IDs `combat-opt-0` through `combat-opt-3`; radio input value is String(val), label textContent is String(val)
    - Note: questionObj.question text is NOT shown in the combat panel — Phase 4 combat questions show only the options, not the text (the enemy flavor text provides context); if the product decision changes in Phase 6, a `<p id="combat-question-text">` can be added

    `updateHP(enemyHP, playerHP)`:
    - Reads `_currentEnemyMaxHP` for enemy percentage; reads `CONFIG.DUNGEON.PLAYER_HP` for player max
    - Clamps: `Math.max(0, Math.min(100, (value / max) * 100))`
    - Sets `document.getElementById('enemy-hp-fill').style.width = enemyPct + '%'`
    - Sets `document.getElementById('player-hp-fill').style.width = playerPct + '%'`

    `showDamageNumber(targetEl, value, isHit)`:
    - Creates a `<span>` with className `'damage-number ' + (isHit ? 'damage-number--hit' : 'damage-number--damage')`
    - Sets span.textContent to `(isHit ? '-' : '+') + value` (minus sign for damage dealt to enemy; plus for... actually: player TAKES damage = red, so use '-' for enemy damage too; both use '-' to show HP loss from the perspective of the target)
    - Actually: isHit=true means player HIT the enemy (enemy takes damage, show green), isHit=false means player took a hit (player takes damage, show red). Use textContent `-` + value for both (both represent HP loss to a target).
    - Appends span to targetEl (the parent must have position: relative — set in CSS in Plan 01)
    - Attaches animationend listener with `{ once: true }` to call `span.remove()`
    - Safety fallback: `setTimeout(() => { if (span.parentNode) span.remove(); }, 600)` — 600ms > 400ms animation duration to catch missed animationend events (RESEARCH.md Pitfall 3)

    `showFeedback(text)`:
    - Sets `document.getElementById('combat-feedback').textContent = text`
    - Does NOT set a clear timeout — feedback is cleared when renderCombatQuestion() runs next

    `renderFloorComplete(summary)` — accepts `{ floor, enemiesDefeated, xpEarned, hpRemaining }`:
    - Sets `#floor-summary-headline` textContent to "Floor " + summary.floor + " Cleared!"
    - Sets `#floor-summary-enemies` textContent to "Enemies defeated: " + summary.enemiesDefeated
    - Sets `#floor-summary-xp` textContent to "XP earned: " + summary.xpEarned
    - Sets `#floor-summary-hp` textContent to "HP remaining: " + summary.hpRemaining
    - Does NOT call App.transition() — caller (CombatInputHandler or Phase 5) handles screen transitions

    `renderLoot(item)` — accepts `{ emoji, name }`:
    - Sets `#loot-emoji` textContent to item.emoji
    - Sets `#loot-name` textContent to item.name

    `renderDead()`:
    - No dynamic content to update — the dead panel HTML from Plan 01 is sufficient; this method is a no-op in Phase 4 (signals caller that dead screen is ready)
    - Implementation: empty function body `renderDead() {}` — Phase 5 may extend if needed

    Critical constraints:
    - The module must NOT call `Renderer.*` methods (no cross-contamination with v1 renderer)
    - The module must NOT read from `#question-text`, `#options-list`, or `#question-fieldset` IDs
    - All dynamic text values (enemy name, floor number, etc.) must go through textContent — never innerHTML
    - `parseInt(selectedValue, 10)` radix is set in CombatInputHandler (Plan 03), not here
