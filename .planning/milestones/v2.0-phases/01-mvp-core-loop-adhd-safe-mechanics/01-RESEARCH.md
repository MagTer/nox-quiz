## Summary

This phase delivers the entire v1 product: a complete, playable, single-file HTML game. No external dependencies, no build step, no server. The file embeds all HTML structure, CSS styling (dark grunge aesthetic with SVG grain), and JavaScript (game loop, question engine, XP system, persistence) in one portable artifact that a 12-year-old can open directly in Chrome on Windows.

The technical patterns for this phase are thoroughly established and well-documented. The game loop uses `requestAnimationFrame`, state is managed through closure-based module objects, localStorage handles persistence with versioned JSON and try/catch error handling, and CSS keyframes drive all animations. No novel or experimental approaches are required — every technique is standard web game practice.

The most critical implementation concern is not technical but behavioral: every design decision must protect the ADHD-adjacent user from anxiety. No countdown timer anywhere (including in auto-advance logic that must be invisible to the user), immediate feedback on answer selection (<300ms), and a clean single-path experience with zero menus or mode choices. The 1-second auto-advance delay is a UX breathing room mechanism, not a visible timer.

**Primary recommendation:** Implement as a single `math-lab.html` file with six closure-based modules (Config, PersistenceStore, PlayerState, QuestionSelector, Renderer, GameLoop) plus a top-level App init. Build in dependency order: Config → PlayerState → PersistenceStore → Renderer → QuestionSelector → InputHandler → GameLoop → App.

### Pattern 1: Closure Module (Namespace Pattern)

**What:** Each logical component is an IIFE that returns a public API; internal state is private via closure.

**When to use:** Single-file apps that need modularity without build tools. Avoids global variable pollution.

**Example:**
```javascript
// [ASSUMED] — standard vanilla JS pattern; widely documented
const PlayerState = (() => {
  let xp = 0;
  let level = 1;
  // EWMA per table: smooth out lucky streaks
  const accuracy = { 1:0.5, 2:0.5, 3:0.5, 4:0.5, 5:0.5,
                     6:0.4, 7:0.4, 8:0.4, 9:0.4 };
  // Sliding window: last 10 answers per table for mastery check
  const history = {};  // { table: [true, false, true, ...] }

  return {
    getXp()    { return xp; },
    getLevel() { return level; },

    addXp(points) {
      xp += points;
      const threshold = XpCalculator.getLevelThreshold(level);
      if (xp >= threshold) {
        xp -= threshold;  // carry over surplus XP
        level++;
        return true;      // signal level-up to caller
      }
      return false;
    },

    // Exponential weighted moving average — recent answers weighted more
    updateAccuracy(table, isCorrect) {
      const prev = accuracy[table] ?? 0.5;
      accuracy[table] = prev * 0.85 + (isCorrect ? 1 : 0) * 0.15;
      // Update sliding window for mastery check
      if (!history[table]) history[table] = [];
      history[table].push(isCorrect);
      if (history[table].length > 10) history[table].shift();
    },

    getAccuracy(table)  { return accuracy[table] ?? 0.5; },

    // Mastery = 80% correct over last 10 answers for this table
    isMastered(table) {
      const h = history[table] ?? [];
      if (h.length < 10) return false;
      return h.filter(Boolean).length / h.length >= 0.80;
    },

    toJSON() {
      return { xp, level, accuracy, history };
    },
    fromJSON(data) {
      xp       = data.xp       ?? 0;
      level    = data.level    ?? 1;
      Object.assign(accuracy, data.accuracy ?? {});
      Object.assign(history,  data.history  ?? {});
    }
  };
})();
```

### Pattern 2: XP & Level Curve

**What:** Level thresholds follow `threshold(n) = 200 × 1.3^(n-1)`. XP surplus carries over to the next level.

**When to use:** All XP reward events.

```javascript
// [ASSUMED] — pure math, no library needed
const XpCalculator = (() => {
  const BASE   = 200;   // XP required for Level 1 → 2  (locked decision)
  const MULT   = 1.3;   // per-level multiplier           (locked decision)
  const XP_EASY = 10;   // tables 1–5                     (locked decision)
  const XP_HARD = 20;   // tables 6–9                     (locked decision)

  return {
    getLevelThreshold(level) {
      return Math.round(BASE * Math.pow(MULT, level - 1));
    },
    calculateXp(table) {
      return table >= 6 ? XP_HARD : XP_EASY;
    }
  };
})();

// Level thresholds:  L1→2: 200 XP, L2→3: 260 XP, L3→4: 338 XP, L4→5: 440 XP...
```

### Pattern 3: Weighted Question Selection

**What:** Tables 6–9 get higher base weight. Tables below 60% accuracy get 1.5× boost. Tables at 80% mastery over 10 answers get reduced weight.

```javascript
// [ASSUMED] — weighted random selection pattern
const QuestionSelector = (() => {
  const HARD_TABLES = [6, 7, 8, 9];
  const EASY_TABLES = [1, 2, 3, 4, 5];

  const calculateWeights = (playerState) => {
    const weights = {};
    HARD_TABLES.forEach(t => {
      const acc = playerState.getAccuracy(t);
      let w = Math.pow(1 - acc, 1.5);
      if (acc < 0.60) w *= 1.5;                // struggling boost (locked decision)
      if (playerState.isMastered(t)) w *= 0.3; // mastered — reduce drilling
      weights[t] = w;
    });
    EASY_TABLES.forEach(t => {
      const acc = playerState.getAccuracy(t);
      weights[t] = Math.pow(1 - acc, 0.8) * 0.3; // ~30% confidence mix
    });
    return weights;
  };

  const weightedRandom = (weights) => {
    const total = Object.values(weights).reduce((a, b) => a + b, 0);
    let roll = Math.random() * total;
    for (const [table, w] of Object.entries(weights)) {
      roll -= w;
      if (roll <= 0) return parseInt(table, 10);
    }
    return HARD_TABLES[0]; // fallback
  };

  // Distractor strategy: nearby multiples of same multiplier (locked decision)
  // ±1 and ±2 variants of the same table, plus one plausible other-table answer
  const generateDistractors = (answer, table, multiplicand) => {
    const distractors = new Set();
    [-2, -1, 1, 2].forEach(d => {
      const v = table * (multiplicand + d);
      if (v > 0 && v !== answer) distractors.add(v);
    });
    // Add a plausible wrong-table answer
    const wrongTable = HARD_TABLES.find(t => t !== table) ?? 7;
    distractors.add(wrongTable * multiplicand);

    const pool = [...distractors].filter(v => v > 0 && v !== answer);
    // Pick 3 unique distractors
    const chosen = [];
    while (chosen.length < 3 && pool.length > 0) {
      const idx = Math.floor(Math.random() * pool.length);
      chosen.push(pool.splice(idx, 1)[0]);
    }
    return chosen;
  };

  // Fisher-Yates shuffle
  const shuffle = (arr) => {
    for (let i = arr.length - 1; i > 0; i--) {
      const j = Math.floor(Math.random() * (i + 1));
      [arr[i], arr[j]] = [arr[j], arr[i]];
    }
    return arr;
  };

  return {
    selectNext(playerState) {
      const weights     = calculateWeights(playerState);
      const table       = weightedRandom(weights);
      const multiplicand = Math.floor(Math.random() * 10) + 1;
      const answer      = table * multiplicand;
      const distractors = generateDistractors(answer, table, multiplicand);
      const options     = shuffle([answer, ...distractors]);

      return { table, multiplicand, answer, options,
               question: `${table} × ${multiplicand}` };
    }
  };
})();
```

### Pattern 4: localStorage Persistence with Versioning

**What:** All state serialized as versioned JSON; every write wrapped in try/catch; save triggered on every answer + visibilitychange.

```javascript
// [ASSUMED] — localStorage versioning is standard best practice
const PersistenceStore = (() => {
  const KEY     = 'mathlab_save_v1';   // locked in CONTEXT.md
  const VERSION = 1;

  const defaults = () => ({
    version:  VERSION,
    xp:       0,
    level:    1,
    accuracy: { 1:0.5, 2:0.5, 3:0.5, 4:0.5, 5:0.5,
                6:0.4, 7:0.4, 8:0.4, 9:0.4 },
    history:  {}
  });

  return {
    load() {
      try {
        const raw = localStorage.getItem(KEY);
        if (!raw) return defaults();
        const data = JSON.parse(raw);
        // Future: if (data.version < VERSION) migrate(data)
        return data;
      } catch (e) {
        console.warn('Save load failed, using defaults:', e);
        return defaults();
      }
    },

    save(playerState) {
      try {
        localStorage.setItem(KEY, JSON.stringify({
          version: VERSION,
          ...playerState.toJSON()
        }));
      } catch (e) {
        if (e.name === 'QuotaExceededError') {
          console.warn('localStorage full — progress may not save');
        }
      }
    }
  };
})();

// Wire up save-on-hide
document.addEventListener('visibilitychange', () => {
  if (document.hidden) PersistenceStore.save(PlayerState);
});
window.addEventListener('beforeunload', () => {
  PersistenceStore.save(PlayerState);
});
```

### Pattern 8: Answer Feedback (Correct / Wrong)

**What:** On answer click: disable all options, add CSS class to show result, auto-advance after 1 second.

  options.forEach(opt => opt.parentElement.classList.add('disabled'));

### Anti-Patterns to Avoid

- **setTimeout-based countdown timer anywhere:** No progress bar that depletes over time, no "answer fast!" prompt, no time-based scoring. The 1s auto-advance is a delay, not a visible timer to the user.
- **Global variables:** All state inside closure modules. No `let xp = 0` at top level.
- **DOM queries inside rAF loop:** Cache all `getElementById` results at init; never query in render().
- **Tightly coupled state + UI in the same function:** `onAnswerClick` only mutates state; `render()` reads state and updates DOM — never mix them.
- **Submitting on radio change:** Attach click handler to the label/input; use a submit button OR auto-advance after delay. Never `<form onsubmit>` that fires on radio change — it bypasses the 1s feedback window.
- **Same 3 distractors every time:** Generate fresh distractors per question; do not cache a static wrong-answers list per problem.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Weighted random selection | Custom probability tree | Simple weighted roll loop (as shown above) | Fisher-Yates + weights-array is 10 lines; O(n) is fine for 9 tables |
| Shuffle answer options | Naive `sort(() => Math.random() - 0.5)` | Fisher-Yates shuffle (as shown above) | `sort` with random comparator has statistical bias; Fisher-Yates is uniform |
| CSS grain texture | PNG/JPG image files or canvas render | SVG `feTurbulence` data-URI (as shown above) | Image adds file size and breaks offline; canvas re-renders every frame |
| XP bar rendering | `<progress>` element | Discrete segment divs | `<progress>` can't be styled segmented without hacks; custom divs are more controllable |
| WCAG contrast checking | Manual eyeballing | Use [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/) before finalizing colors | Human perception of contrast is unreliable; tool takes 5 seconds |
| Random number seeding | Custom PRNG | `Math.random()` | No reproducibility needed; `Math.random()` is sufficient for question shuffling |

**Key insight:** This is a linear, turn-based game with no physics. The "hard" problems — ADHD-safe adaptive weighting, XP curves, grain textures — all have 10–20 line vanilla solutions. Reach for browser natives before inventing abstractions.

## Common Pitfalls

### Pitfall 1: The Hidden Timer
**What goes wrong:** Developer adds a `setTimeout` or rAF counter that visually counts down in any form — a progress bar draining, a color change over time — thinking it adds urgency. The user immediately perceives pressure. ADHD context: cortisol spike → working memory collapse → wrong answers → app abandoned.

**Why it happens:** It feels like a "game-y" feature. The 1s auto-advance is already a timing mechanism so the temptation to expose it is high.

**How to avoid:** The auto-advance 1s delay must be invisible to the user. No visual element changes during that 1 second except the feedback state (correct/wrong colors). No shrinking bar, no countdown text, no spinner.

**Warning signs:** Any CSS `transition` on an element that changes during the 1s feedback window that could be perceived as a timer.

### Pitfall 2: Answer Position Bias
**What goes wrong:** Correct answer clusters in one position because distractors are generated without shuffling, or `shuffle` uses `sort(() => Math.random() - 0.5)` (which is statistically biased).

**Why it happens:** `sort` with a random comparator is the common "quick shuffle" that most developers reach for. It has non-uniform distribution.

**How to avoid:** Use Fisher-Yates (Knuth) shuffle as shown in Pattern 3. Verify across 50+ generated questions that position distribution is ~25% each.

**Warning signs:** After 20 questions, correct answer has appeared in position 0 more than 8 times.

### Pitfall 3: Quota-Silent localStorage Failures
**What goes wrong:** `localStorage.setItem()` throws `QuotaExceededError` silently (no crash, just a thrown exception), XP is lost, user opens app next day at Level 1.

**Why it happens:** No try/catch around the setItem call. This is the single most common single-file game bug.

**How to avoid:** Every `setItem` wrapped in try/catch. On catch, log to console and optionally show a one-time non-blocking toast. The game continues in memory; nothing crashes.

### Pitfall 4: Distractor Collision with Correct Answer
**What goes wrong:** `generateDistractors(42, 6, 7)` generates a distractor that equals 42 (the answer). Duplicate appears in options list.

**Why it happens:** Math like `6 × (7 + 1) - 6` can occasionally resolve to the answer value. Adjacent multiplicands can collide.

**How to avoid:** Filter `v !== answer` in every distractor candidate before adding to pool. The code example in Pattern 3 already does this.

### Pitfall 5: Level-Up XP Surplus Lost
**What goes wrong:** Player has 195 XP, gets +20 XP (hard table), reaches 215 XP. Threshold is 200. Level increments to 2 but XP resets to 0 instead of carrying 15 XP surplus.

**Why it happens:** `xp = 0` after level up instead of `xp = xp - threshold`.

**How to avoid:** `xp -= threshold` not `xp = 0` in the addXp level-up branch. Example in Pattern 1 demonstrates this.

### Pitfall 6: DOM Queries Inside rAF Render
**What goes wrong:** `document.getElementById('xp-bar')` called every frame. Profiler shows constant layout thrashing; UI jank after 5 minutes.

**Why it happens:** Inline queries feel natural but browsers must walk the DOM tree each time.

**How to avoid:** Cache all element references at `App.init()` into a `const DOM = { xpBar: ..., ... }` object. Renderer reads from DOM cache, never re-queries.

### Pitfall 7: SVG Grain at Wrong Opacity
**What goes wrong:** `feTurbulence` grain at 15–20% opacity becomes visually overwhelming; distracting for ADHD. At <2% it's invisible and pointless.

**Why it happens:** Opacity tuned on developer's monitor which may have different gamma. Looks fine on their screen, harsh on the target Windows laptop.

**How to avoid:** Store grain opacity in a CSS custom property `--grain-opacity: 0.05` (locked decision: ~5%). Test visually on a Windows screen in a bright-room environment. The CSS variable makes it trivial to adjust.

### Pitfall 8: Radio Auto-Submit on Selection
**What goes wrong:** `input[type=radio]` wired to immediately fire the answer validation on `change` event, skipping the feedback window. User accidentally touches the wrong option while scrolling and it counts as wrong.

**Why it happens:** `addEventListener('change', validate)` is the first pattern most developers reach for.

**How to avoid:** Wire click handlers to the labels (or use a separate confirm button if desired). In the auto-advance model, click on a label fires `handleAnswer()` which triggers the 1s delay before advancing — the radio input change is not the trigger; the label click is.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `setInterval` for game loops | `requestAnimationFrame` | ~2012 (browser standard) | rAF pauses when tab hidden; synced to vsync; no jank |
| `sort(() => Math.random() - 0.5)` for shuffle | Fisher-Yates (Knuth) shuffle | Documented bias since ~2007, widely known by 2015 | Uniform distribution; no position clustering |
| PNG/JPG textures for grunge effects | SVG `feTurbulence` data URI | ~2015 (CSS filter support matured) | Zero file size; resolution-independent; no network request |
| Inline `onclick=` attributes | `addEventListener` in JS module | ES5 era | Separates concerns; no global scope pollution |
| `window.onload = function()` | `DOMContentLoaded` event | Standard since 2009 | Fires before images load; faster init |

**Deprecated/outdated:**
- `setInterval` for any animation or game timing: replaced by `requestAnimationFrame`
- `localStorage.setItem` without try/catch: undefined behavior when quota exceeded; always wrap
- `sort(() => Math.random() - 0.5)` for shuffling: statistically biased; use Fisher-Yates

## Open Questions

1. **Is `#00ff88` text on `#0a0a0a` WCAG AA compliant?**
   - What we know: Neon green on near-black typically has high luminance contrast; primary text is `#e8e8e8` which is safe.
   - What's unclear: `#00ff88` as text (e.g., "LEVEL UP!" text color) vs. as decorative color.
   - Recommendation: Run WebAIM contrast check as the first verification task in Wave 0. If it fails as text, use it as glow/shadow only; switch text to `#ffffff` or `#e8e8e8`.

2. **Should auto-advance be triggered by label click or by a "Next" button?**
   - What we know: Locked decision says "auto-advance after 1 second feedback delay."
   - What's unclear: Whether the user should be able to override the 1s delay (click anywhere to advance early).
   - Recommendation: Implement click-anywhere-to-advance-early as a quality-of-life feature: if user clicks again during the 1s window, advance immediately. This is not a timer the user sees — it just lets impatient users move faster.

3. **Font choice for the "bold/grunge" direction?**
   - What we know: Claude has discretion on font choice.
   - What's unclear: Web-safe bold fonts vs. embedding a font in base64.
   - Recommendation: Use `font-family: 'Impact', 'Arial Black', sans-serif` — pre-installed on Windows; zero file size. If a more distinctive grunge font is desired, embed a small subset as a base64 data URI (adds ~10–20 KB).
