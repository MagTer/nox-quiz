    Add a new section **"6b. Motion design rules (HARD — read before authoring ANY mover, Phase 36+)"**, directly after section 6a's mover preview. Write these three as HARD rules, each with its reasoning, in the doc's existing voice:

    1. **A checkpoint before every mover.** A moving platform or patrol section must sit behind a checkpoint, so that a mistimed jump costs at most the approach, never real progress. This is section 5's respawn policy extended to motion.
    2. **A missed moving platform means WAIT, never death.** If the player misses a mover, the level must leave them somewhere they can simply wait for it to come back (a ledge, a safe floor below, the previous tier) — never a pit that kills, never a state they cannot recover from without a respawn. Movers ping-pong; the player must be able to ping-pong with them. There is no game-over anywhere in this game, and a mover must not become the first one.
    3. **Patrollers carry ZERO hurt wiring.** A patrolling enemy is a cosmetic hazard on the existing respawn class: touching one returns the player to their checkpoint, full stop. No damage, no health, no lives, no score penalty, no failure construct of any kind. Stationary math-blocker enemies keep their existing role and must stay visually distinct from patrollers.

    Tie the three to the enforcement that already exists (SAFE-01, `check-safety.sh`'s no-timer/no-punishment scan over `src/`, and the validator's `mover-reachability` worst-case-extreme rule from Phase 30, including its documented rightward-travel-only limitation in section 6a) so an author can obey them without reading the shell gate. State explicitly that these rules exist BEFORE the first mover is authored, and that Phase 36 is the first phase permitted to place one.

    Add a new HARD rule for coins — the bug class this phase just closed. Put it where coins naturally live in the doc (a new short section after section 3, or extend section 2/3 — your call, keep the doc's structure coherent):
    - **HARD (coin reachability):** every coin must be reachable — `node scripts/validate-levels.mjs`'s `coin-reachability` row, plus `node scripts/audit-coins.mjs`'s in-engine witness replay, must both be green.
    - Explain the model in one paragraph, because getting it wrong is how the bug survived four milestones: a coin is NOT a zero-width point like a secret alcove. `build.js` gives it a 32x32 `area()`, and the 16x32 player collects it MID-ARC by passing through it — so the reachable target is a 48x64 box of player top-left positions, not a landing spot. Probing coins through the secret-alcove point model OVER-REPORTS; the alcove model asks a landing question, the coin model asks a fly-through question.
    - Practical authoring guidance: walking height over a surface is roughly `surfaceY - 56`; an arc coin must sit within the calibrated maxRise (88.331px) of the highest reachable surface beneath it, or no jump touches it at any x.
    - Note the static model's deliberate obstruction blindness (it does not know about platform undersides or blockers — see section 3's ceiling-bonk rule) and that `audit-coins.mjs` is the in-engine arbiter that covers it.
