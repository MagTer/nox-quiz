# Phase 34 Plan 07: Make the Verification Harness Bidirectional

## Objective

Teach the Playwright traversal harness to drive the player **LEFT as well as RIGHT**, so a
switchback climb can be navigated from spawn. Today the driver can only hold `ArrowRight`,
so `scripts/browser-boot.mjs` is RED on level-08 and level-08's secret alcove reports
`triggered: false` in `scripts/audit-phase21-mechanics.mjs`.

## Why this matters

**Nothing automated currently proves level-08 is navigable from spawn.** `audit-coins.mjs`
TELEPORTS the player onto each tier, so it proves coins are collectable *on* the tiers, not
that the climb can be *reached*. Only a human's play proves level-08 completable — a red gate
we would otherwise be quietly agreeing to ignore.

Phase 34.5 is about to REBUILD ALL 8 LEVELS at twice the length. Without a bidirectional
driver, none of those rebuilt levels can be automatically proven completable. Phase 36's
moving platforms and patrols need it too. This is the gate that makes the rebuild verifiable.

The irony worth internalising: plans 34-01/34-02 deliberately made the COIN model bidirectional,
citing this very switchback (see `bestWitnessToCoin`'s own header: *"level-08's Phase-34
switchback climb reverses direction twice, so a rightward-only coin model would fabricate false
HARD-FAILs on it"*). Nobody extended the same reasoning to the DRIVER.

## Root cause (confirmed, not assumed)

1. **`scripts/lib/mechanic-drive.mjs :: driveToXPlanned`** presses `ArrowRight` ONCE and HOLDS
   it for the whole drive (`page.keyboard.down("ArrowRight")` before the loop, `up` in the
   `finally`). Its progress model is monotonically-increasing `maxX`. A leftward leg therefore
   reads as a stall, and the bot walks off the tier and dies.
2. **`scripts/lib/route-planner.mjs :: planTakeoffs`** *finds* the switchback path in the graph
   (all 10 nodes, including the leftward T3→T4 hop) but emits no USABLE takeoff for it:
   `computeMountTakeoffX` computes `x = to.xStart - offset` — a RIGHTWARD mount mark — which for
   a leftward hop clamps to `from.xStart + 4` and points the jump the wrong way entirely.

The static reachability graph is already correct: `canReach` explicitly handles
`toNode.xEnd <= fromNode.xStart` (leftward), and `buildGraph` tests every ordered pair BOTH
ways. **The graph knows the path exists; the driver cannot follow it.**

<task type="auto">
  <name>Task 2 — mechanic-drive: leg-sequenced, non-monotonic driver</name>
  <files>scripts/lib/mechanic-drive.mjs</files>
  <what>
  Rewrite `driveToXPlanned`'s control loop around LEGS instead of a single held ArrowRight.

  - **Held direction** is derived every grounded tick from the CURRENT LEG, which is itself
    derived from live player state (which surface the feet are on) — never from a stored
    cursor. That makes it self-healing across deaths exactly as the current `consumed`-set
    reset is. While airborne, the last direction is HELD (jump arcs need sustained input).
  - **Takeoff firing becomes direction-relative**: `along = t.dir * (x - t.x)`; fire when
    `0 <= along <= fireWindow`. For `dir === +1` this is byte-identical to today's
    `x >= t.x && x <= t.x + window`.
  - **Approach/backup phase** (the load-bearing new behavior): if the current leg's takeoff is
    pending and `along > fireWindow` (the player has OVERSHOT the launch mark), hold `-dir` to
    walk back onto the mark, then turn and fire. This is what flies BOTH reversals:
      * REVERSAL 1 (T3→T4, up-LEFT): land at T3's left end, walk RIGHT past T4's far edge onto
        T3's runway, turn around, jump back up-left. Emergent from the general model.
      * REVERSAL 2 (T4→T5, up-RIGHT): land at T4's RIGHT end (~2913), walk LEFT to ~2802 for
        run-up, sprint right and jump. Also emergent — this is exactly the maneuver level-08's
        own descriptor comment documents.
    Backup engages ONLY for route (leg) takeoffs, never for spike takeoffs, and is clamped to
    the launch surface's own span so it can never walk into a gap.
  - **Progress model becomes NON-MONOTONIC**: progress is "advanced along the route"
    (`legIndex`, then `dir * x` within the leg), never "x got bigger". The stall timer resets on
    a leg advance, a new per-leg max, a takeoff fire, or a death.
  - **Death detection becomes direction-agnostic**: a position WARP (`hypot(dx, dy) > 200px`
    between consecutive ~55ms samples) instead of `maxX - x > 250`. A leftward leg's legitimate
    ~13px/tick backward walk can never be misread as a respawn.
  - **Arrival**: once on the target node, walk toward `targetX` from whichever side — including
    LEFTWARD (level-08's alcove at x:2650 sits LEFT of T4's landing spot at ~2913, which is
    precisely why it reports `triggered: false` today).
  - Public contract (`driveToXPlanned`, `resolveIfBoxed`, `deriveEncounters`,
    `driveAndDetectAlcove`) is UNCHANGED, so `browser-boot.mjs`, `audit-phase21-mechanics.mjs`,
    `audit-retry.mjs`, `screenshot-phase26.mjs` and `screenshot-phase34-climb.mjs` keep working.
  </what>
  <verify>`node scripts/browser-boot.mjs` — level-08 reaches goal.x:3460; levels 01–07 unchanged.</verify>
  <done>All 8 levels drive to goal; level-08's alcove triggers.</done>
</task>
