### CR-01: `key-lock-check.mjs` falsely reports a key reachable when it sits just past its own lock

**Files modified:** `scripts/lib/key-lock-check.mjs`
**Commit:** 7b890a8
**Applied fix:** Added a lock-aware wrapper, `bestMarginToPointExcludingBand`,
around the shared `bestMarginToPoint` (reachability.mjs, left untouched since
it's shared with the alcove/coin/mover checks). The wrapper drops any
candidate source node that sits on the OPPOSITE side of the lock's band from
the key before delegating to the original point-reach model — since
`clipNodesAtBand` already guarantees every surviving node is unambiguously on
one side of the band, this is sufficient to reject every hop whose straight
line would pass through the lock's own solid apex-tall wall. Added self-test
case (f) reproducing the reviewer's exact false-PASS window (key placed 48px
past `bandHi`, well inside the reported ~10-110px window) — now correctly
HARD-FAILs. All five prior self-test cases (a-e: same-run behind-lock, legit
reachable, platform-over-band guard, multi-pair keyId, no-keys/locks
never-throw) still pass unchanged.

**Verification:**
- `node scripts/lib/key-lock-check.mjs` self-test → PASS (6 cases, all green).
- `node scripts/validate-levels.mjs --fixture scripts/fixtures/bad-level-key.js` → still exit 1 (RED-first proof holds).
- `node scripts/validate-levels.mjs` on real levels → exit 1, exactly the same 13 known-deferred HARD-FAILs (all `coin-reachability`/`headroom`, none `key-lock`), level-02's key-lock row unchanged at `WARN` (key/lock ~3000px apart). Zero new key-lock HARD-FAILs.

### WR-01: `keyHeld` had no keyId identity but the validator models keyId pairs

**Files modified:** `src/levels/build.js`, `src/mechanics/key.js`, `src/scenes/game.js`
**Commit:** 41fecee
**Applied fix:** Threaded `keyId` end-to-end so the runtime matches the
validator's data model:
- `build.js` stashes `l.keyId ?? null` / `k.keyId ?? null` directly on the
  lock blocker and key entity (same direct-property-assignment idiom already
  used for `blocker.panelObj`).
- `key.js`'s `wireKey` passes the touched key's id through `onPickup(keyId)`
  and reads the collided lock's id via `hasKey(keyId)`, rather than
  zero-argument callbacks.
- `game.js` replaced the single `let keyHeld = false` boolean with a
  closure-local `const heldKeyIds = new Set()`, normalizing a missing keyId to
  a shared `KEY_DEFAULT_ID` sentinel so today's single-pair, no-id level-02
  case behaves EXACTLY as before (any key opens the one lock). A future
  multi-lock level with distinct `keyId`s will now be correctly gated
  per-pair instead of any key opening every lock.

**Verification:**
- `bash scripts/check-safety.sh`, `bash scripts/check-import-safety.sh` → PASS (no timer/scheduler introduced, no module-level run state, a727c13-safe engine-global usage unchanged).
- `node scripts/audit-key-lock.mjs` (isolated) → PASS — real driven player collects the key then opens the lock in-engine, confirming the id-normalization path is exercised end-to-end for the current no-id level-02 pair.
