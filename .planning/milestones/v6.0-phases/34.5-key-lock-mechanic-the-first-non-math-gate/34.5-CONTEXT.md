**Gathered:** 2026-07-15

<domain>
## Phase Boundary

Build the key/lock mechanic as CODE — a `src/mechanics/key.js` seam, `geometry.keys`/`geometry.locks`
descriptor fields, a HARD softlock validator check, and real in-engine driven-player audit coverage —
BEFORE any level is authored to use it (Phase 34.6 rebuilds all 8 levels and will place key-locks then).

This phase is the game's FIRST non-math gate. It knowingly reverses SEED-001's "no new play mechanics"
clause (user's explicit override, 2026-07-15). The end-of-level math gate remains the final challenge;
the key-lock is an ADDED barrier, not a replacement.

IN SCOPE: the mechanic module, descriptor fields, builder wiring, HUD key-held indicator, the HARD
softlock validator check + RED-first fixture, a driven-player audit, and ONE throwaway key-lock placed
on level-02 purely to prove the pipeline in-engine.

OUT OF SCOPE: authoring key-locks across the 8 levels (that is Phase 34.6), any geometry rebuild, art.

</domain>

<decisions>
## Implementation Decisions

### Pacing — math density (the flagged open question, user-decided)
- **ADD a 4th barrier — do NOT replace the math door.** Math density stays LOCKED at 3 math challenges
  per level (1 mid-level math door + 1 enemy + 1 end goal math-gate). The key-lock is a NEW *non-math*
  gate layered on top. This preserves the "3 math challenges" lock; the only lock the user reversed is
  SEED-001's no-new-mechanics clause, not the math-density lock.
- **The key opens the EXIT** (user chose this over an interior barrier). The lock guards the approach to
  the end of the level — she must have collected the key before she can finish. **Synthesis to preserve
  the thesis:** the lock is placed as the penultimate barrier guarding the corridor to the goal; the
  end-of-level MATH GATE remains the literal final challenge. So the ordering at level end is:
  locked exit-corridor (needs key) → end math gate → LEVEL CLEAR. This keeps "multiplication is the
  final gate to progress" true while making the key feel like it opens the way out.
  - **Load-bearing consequence:** because the lock guards the point of no return, a missed key is the
    worst-case softlock (stuck AT the finish). The HARD reachability check below is therefore mandatory
    and non-negotiable — this is the most softlock-sensitive placement possible, chosen deliberately.
  - **Sequencing trap for the planner:** the end math-gate transitions the scene away (`go("select")`)
    on a correct answer. The key-lock MUST therefore gate BEFORE the math gate can fire, or the lock is
    never evaluated. The lock blocks reaching the goal/math-gate; it is not co-located with it.
- **Optional per-level tool.** The mechanic supports per-level opt-in via presence of `keys`/`locks` in
  a descriptor. It is NOT mandatory on every level. Phase 34.6 decides which levels use it (guidance:
  the even/"intense + vertical" levels benefit most). A level with no `keys`/`locks` behaves exactly as
  today.
- **One key-lock pair per level for now**, but the DATA MODEL supports multiple via an optional `keyId`
  to match a key to its lock. Author one pair per level to keep cognitive load low for a 12-year-old;
  the multi-pair capacity is latent, not exercised yet.

### No-softlock behavior & feel (accepted as recommended)
- **At a lock without the key:** the lock is a solid collider that simply does not open, plus a brief,
  non-blocking on-screen hint (e.g. "Find the key"). NO damage, NO bounce-back, NO reset, NO game-over.
  The remedy is always "go back and get it."
- **The key persists across in-level respawn.** Key-held is run-state (closure-local to the scene) that
  survives a pit-fall / checkpoint respawn. Dropping the key on death would be a soft-punishment AND a
  softlock risk — explicitly rejected.
- **The key is visible and telegraphed.** The key is a visible pickup and the lock is a visibly distinct
  barrier. No hidden/secret keys — aligns with the level-shape brief ("she must always be able to SEE
  where she's going; no hidden routes," 2026-07-15). A hidden key is a softlock risk and is rejected.
- **Feedback:** key pickup uses the secret-alcove idiom (fx burst + chime + rising popup). While the key
  is held, a persistent HUD "key held" indicator is shown so she knows she has it.

### Data model, validator & audit (accepted as recommended)
- **Descriptor fields:** `geometry.keys: [{ x, y }]` and `geometry.locks: [{ x, y }]`, matching the flat
  `{x,y}` idiom already used by `coins`, `checkpoints`, `secretAlcove`. Optional `keyId` on both for
  multi-pair matching (unused for now but part of the shape).
- **Mechanic module:** a NEW `src/mechanics/key.js` wiring BOTH halves — pickup (secretAlcove-style:
  walk-through, no challenge, fire-once closure-local latch, feedback) and lock-open (door.js structure
  MINUS the math challenge: a solid barrier that, on collide, opens+destroys iff key-held, else shows
  the hint). The ONE builder (`src/levels/build.js`) owns entity creation from the descriptor arrays;
  `key.js` only wires collision handlers against tags. a727c13 discipline: engine globals inside the
  function body only.
- **Softlock validator:** a NEW `scripts/lib/key-lock-check.mjs` composed into `validate-levels.mjs`,
  mirroring the existing `over-hole-check.mjs` / `reachability.mjs` composition and the two-tier
  HARD-FAIL/WARN output. It must BFS-prove, per level: every lock's key is reachable from spawn AND lies
  on the spawn-side of its lock (reachable BEFORE the lock is required). Violation = HARD-FAIL that
  increments the failures counter, exactly like coin-reachability. Ship a RED-first fixture under
  `scripts/fixtures/` proving the HARD-FAIL fires (a key placed behind its own lock).
- **In-engine proof vehicle:** add ONE key-lock to level-02 (shortest real level, already has a door) so
  a real driven player (the now-bidirectional Phase-34 harness) picks up the key and opens the lock in an
  `audit-*.mjs`. This is acknowledged throwaway — Phase 34.6 rebuilds level-02 from scratch — but it is
  the only way to get REAL in-engine coverage now, and this milestone has repeatedly shipped bugs past a
  green static model, so a static model alone is insufficient (success criterion 3).

### Claude's Discretion

  later phase), and the precise BFS formulation of "spawn-side of the lock" — all at Claude's discretion,

## Existing Code Insights

### Reusable Assets
- `src/mechanics/secretAlcove.js` — the walk-through, non-challenge, progress-wired pickup pattern the
  KEY half of `key.js` should mirror (closure-local fire-once `Set`, feedback burst/chime/popup, destroy
  on pickup, no pause).
- `src/mechanics/door.js` — the solid-collider-that-blocks structural pattern the LOCK half should mirror
  (WR-03 re-entrancy `busy` guard, destroy collider+panel+glyph BEFORE unfreezing to avoid same-frame
  re-collide softlock) — MINUS the `openChallenge()` call; the lock checks key-held instead.
- `src/levels/build.js` — the ONE descriptor→entities builder; owns creating tagged `key`/`lock`
  entities from the new geometry arrays (and the `?debug=1` overlay for them).
- `scripts/validate-levels.mjs` + `scripts/lib/over-hole-check.mjs` + `scripts/lib/reachability.mjs` —
  the compose-a-lib-check-into-the-orchestrator pattern, the two-tier HARD-FAIL/WARN reporting, and the
  `--fixture <path>` single-descriptor mode the new check + RED-first fixture plug into.
- `scripts/audit-phase21-mechanics.mjs` (driven player) and `scripts/audit-coins.mjs` (reachability
  audit) — the driven-player audit patterns; the Phase-34 bidirectional harness makes driving back for a
  key possible.
- `src/ui/hud.js` — where the persistent "key held" indicator is added (level badge + XP bar today).
- `src/config.js` — ALL tunables (key/lock sizes, HUD indicator, hint) live here; no magic numbers in
  logic modules.

### Established Patterns
- Levels are PURE DATA (`src/levels/level-0N.js`), flat geometry arrays; the registry
  `src/levels/index.js` stays node-importable (no engine refs) so the validator can import it.
- Mechanics wire `player.onCollide(tag, ...)` handlers; a single wiring call handles all entities of a
  tag (Kaplay passes the touched object to the callback).
- No timers/schedulers anywhere in `src/`; delayed effects use `tween().onEnd()` self-clean. Enforced by
  `check-safety.sh`. Run state is closure-local, never module-level.
- Validator scripts are standalone `node` invocations (no test framework, no npm), fast because
  pure-data.

### Integration Points
- `src/scenes/game.js` builds the level and wires mechanics — `wireKey(...)` gets called here alongside
  the existing `wireDoor`/`wireSecretAlcove` calls, threaded with the scene's closure-local key-held
  state and the HUD.
- `scripts/validate-levels.mjs` gains the new lib check in its per-descriptor loop.
- `node scripts/check-assets-manifest.mjs` — any new key/lock sprite path must be declared in
  `src/assets-manifest.js` or the manifest gate fails.

<specifics>
## Specific Ideas

- End-of-level ordering is FIXED by the "key opens the exit" decision: locked exit-corridor (needs key)
  → end math gate → LEVEL CLEAR. The lock is NOT co-located with the math gate; it gates the approach so
  the math gate stays the final act.
- The RED-first fixture is a required deliverable, not optional: a `scripts/fixtures/` level with a key
  sealed behind its own lock, proving `key-lock-check.mjs` emits a HARD-FAIL. Follows the milestone's
  RED-first harness discipline (Phase 30 / headroom gate).
- Reuse the removed-mechanic caution: `collect.js` was deleted in Phase 29 for being a bad fit — the key
  is NOT a collect-the-answer revival; it is a spatial traversal gate, wired like the alcove.
