**One throwaway key-lock pair placed on level-02 (key x:900, lock x:3960) proven softlock-free by the static validator AND proven collectible/openable by a real driven player via a new `scripts/audit-key-lock.mjs` — closing KEY-01/KEY-02 with the SC3 in-engine proof this milestone's own history (sawtooth floor, grey ground, ceiling-bonk coins) says a green static model alone cannot provide.**

## Decisions Made

- **Moved the key from x:760 (RESEARCH's suggested position) to x:900.** Found by actually running `node scripts/browser-boot.mjs`, not by re-reading the static model: x:760 sits inside the jump arc that clears the 520..700 gap onto floor-1, so a driven player jumping that gap lands past x:790 and never touches down on a ground-level trigger at x:760. `validate-levels.mjs`'s reachability model is deliberately arc/landing-blind (documented in `reachability.mjs`'s own header, the same blindness class `audit-coins.mjs` exists to catch for coins), so it still reported x:760 as reachable — this is precisely the class of bug this plan's SC3 mandate (a real in-engine drive, not just a green static model) exists to catch. x:900 sits comfortably past every observed landing spot and well before the next takeoff (~x:1200), squarely in the walked stretch between them; confirmed by both a direct `driveToXPlanned` walk-through and the full `audit-key-lock.mjs` run.
- **`audit-key-lock.mjs` resolves intervening door/enemy encounters before driving to the lock**, rather than driving key -> lock in one shot as a first read of the plan's Finding 5 pseudocode suggested. `driveToXPlanned`'s `triggered` return is a generic "did any challenge open beyond baseline" signal — driving straight toward the lock (x:3960) stalls the instant the door's challenge (x:1540) opens, since the driver has no way to know that specific challenge isn't the caller's target. This is a property of the shared `mechanic-drive.mjs` driver used across every audit script in this tree, not a level-02-specific defect, so the fix lives in this audit's own drive sequence — mirroring `browser-boot.mjs`'s own "resolve every remaining encounter on the path first" loop, which exists for the identical reason.
- **`scripts/smoke-progress.mjs`'s LVL-02 geometry regression assertion needed updating**, not loosening: it is a byte-for-byte `deepEqual` against level-02's expected descriptor, and it correctly went RED the moment Task 1 added `keys`/`locks` — proof the regression test does its job. The fix adds the same two fields to `expectedGeometry`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Key placement (x:760) sat inside a jump arc's landing zone and was never collected by a driven player**
- **Found during:** Task 1's own verification (`node scripts/browser-boot.mjs`)
- **Issue:** The RESEARCH-suggested key x:760 is geometrically reachable per the static model but physically un-touched by a real driven player clearing the preceding gap — the jump arc lands past it. `browser-boot.mjs` stalled at the lock (never having collected the key) with `far-end-unreachable`.
- **Fix:** Moved the key to x:900 (same floor run, past the landing zone, before the next takeoff). Verified via a direct `driveToXPlanned(door.x)` drive that the key is now collected incidentally en route.
- **Files modified:** `src/levels/level-02.js`, `scripts/smoke-progress.mjs` (regression-test coordinates updated to match)
- **Commit:** `9fe0ac5`

**2. [Rule 1 - Bug] `driveToXPlanned`'s generic `triggered` flag stalls a key->lock drive at an intervening math challenge**
- **Found during:** Task 2's own verification (`node scripts/audit-key-lock.mjs`)
- **Issue:** A direct `driveToXPlanned(page, lock.x, geometry)` call after collecting the key stopped at the door's challenge (x:1540, `reachedX:1524`) because `triggered` is baseline-relative to `get("challenge").length`, not specific to the caller's target.
- **Fix:** Added a resolution loop (`deriveEncounters` + `resolveIfBoxed`, mirroring `browser-boot.mjs`) that resolves every door/enemy encounter strictly between the key and the lock before the final drive to the lock.
- **Files modified:** `scripts/audit-key-lock.mjs`
- **Commit:** `70950a6`

**3. [Rule 3 - Blocking] `check-progress.sh`'s LVL-02 geometry regression test failed after Task 1's descriptor change**
- **Found during:** Full-suite verification after Task 1
- **Issue:** `scripts/smoke-progress.mjs` deep-equals `getLevel("level-02").geometry` against a hardcoded `expectedGeometry` that predates the new `keys`/`locks` fields — legitimately went RED.
- **Fix:** Added the matching `keys`/`locks` entries to the test's `expectedGeometry`.
- **Files modified:** `scripts/smoke-progress.mjs`
- **Commit:** `70950a6` (bundled with Task 2 — discovered while re-verifying the full gate suite after Task 1's placement fix, landed alongside the audit script since both are same-scope hardening of this plan's own deliverable)
