# Phase 34.5: Code Review Report

**Reviewed:** 2026-07-15T00:00:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

The in-engine mechanic (`src/mechanics/key.js`) is clean: closure-local fire-once
latches, no module-level run state, destroy-before-continue ordering, and a
self-cleaning `tween().onEnd()` hint with zero timers — no a727c13 or
no-timer violations. `check-import-safety.sh`'s scoped scan already includes
`key.js` in both the existence and negative-grep lists, and the gate is
genuinely RED-first-provable (`scripts/fixtures/bad-scene.js` still trips it).
`scripts/lib/reachability.mjs`'s only change is exporting two
already-implemented functions (`bfsWithPathMargin`, `bestMarginToPoint`) — confirmed
via `git diff` to be a two-line, behavior-free change.

The one serious defect is in `scripts/lib/key-lock-check.mjs` itself — the
module whose own header calls a false PASS here "the worst failure this
project can make." Its node-splitting/edge-cutting logic (`clipNodesAtBand` /
`buildLockCutGraph`) is sound and was verified by direct reproduction against
several adversarial geometries (a lock at a run boundary, a lock floating
entirely inside a gap between two floor runs, keyId-based multi-pair
matching). But the function it delegates to for the actual key-vs-lock
distance test, `bestMarginToPoint` (imported unmodified from
`reachability.mjs`, originally built for secret-alcove points that have no
solid collider), has **no awareness that the lock's own band is a real, solid,
apex-tall wall**. A key placed anywhere from ~10px to ~100px past its own
lock's band is reported reachable (PASS/WARN) instead of HARD-FAIL, directly
contradicting the file's own "sound at ANY placement" claim — reproduced
below with concrete numbers. This does not affect the currently-shipped
level-02 pair (key x:900, lock x:3960, ~3000px apart, well outside the
false-PASS window), but it is a real, silent softlock-shipping hole in the
gate itself, exactly the failure mode this phase exists to prevent — the
next lock a human places anywhere near this proximity to its key ships an
undetected dead end.

A second, independent issue: the static validator explicitly models and
self-tests multiple *distinct* key-lock pairs matched by `keyId` (case (d) in
`key-lock-check.mjs`'s self-test), but the real in-engine mechanic
(`src/mechanics/key.js` / `src/scenes/game.js`) has no `keyId` concept at
all — `keyHeld` is a single scene-wide boolean. Any future level with two
independently-gated locks would validate cleanly but, at runtime, picking up
*either* key would open *both* locks.

## Critical Issues

### CR-01: `key-lock-check.mjs` falsely reports a key reachable when it sits just past its own lock (bestMarginToPoint has no wall-obstruction awareness)

**File:** `scripts/lib/key-lock-check.mjs:177` (calls into `scripts/lib/reachability.mjs:406-466`, `bestMarginToPoint`)
**Issue:**

`checkKeyLockReachability` correctly severs node-to-node connectivity across a
lock's band (`clipNodesAtBand` + `buildLockCutGraph`'s opposite-side edge
drop, verified sound against a lock at a run boundary and a lock floating in
a gap between two floor runs). But the actual "is the key within reach"
test — `bestMarginToPoint(point, cutNodes, spawnPaths, envelope)` — is
reused **unmodified** from the secret-alcove/point model in
`reachability.mjs`. That model was designed for triggers with no solid
collider (an alcove has no body); it computes whether a point lies within a
jump/fall's raw horizontal reach from *any* spawn-reachable node, with **no
check that the straight-line hop it credits doesn't pass straight through the
lock's own solid apex-tall wall**. Because `bestMarginToPoint`'s "cross-height
gap hop" family (case 3, `reachability.mjs:448-458`) is evaluated per
spawn-reachable *node* — not per *cut graph edge* — it is completely blind to
the same-band exclusion that `buildLockCutGraph` otherwise enforces.

Reproduced directly (floor `{x:0,w:900}`, lock at `x:400` → band `[400,432)`,
`FLOOR_Y - CONFIG.LOCK.H`):

```
key at bandHi+10  (x=442): WARN  — reported "reachable spawn-side of lock" (should be HARD-FAIL)
key at bandHi+32  (x=464): WARN  — same
key at bandHi+50  (x=482): WARN  — same
key at bandHi+80  (x=512): WARN  — same
key at bandHi+120 (x=552): HARD-FAIL — correct (finally past the false-positive window)
```

A key placed anywhere from just past the lock's own 32px footprint up to
roughly 100-110px beyond it (the flat/downward `bestMarginToPoint` reach at
this game's calibrated envelope) is silently reported as obtainable
*before* the lock, when in the real engine reaching it requires literally
passing through the lock's solid, apex-tall collider — i.e. requires the key
already being held. That is precisely the "key sealed behind its own lock"
softlock this whole subsystem exists to catch, and `bad-level-key.js`'s
RED-first fixture (key at `x:600`, 168px past `bandHi`) happens to sit well
outside this window, so the self-test suite never exercised it. The current
level-02 pair (key at x:900, lock's band starting at x:3960) is nowhere near
this window and is unaffected, but nothing in `checkKeyLockReachability` or
`validate-levels.mjs` would catch a future level (Phase 34.6 is explicitly
scoped to add more locks) that places a key a short, natural-looking jump
past its own lock — exactly the kind of placement a human level author would
plausibly choose "to make it feel close, but still definitely locked."

**Fix:** Give `bestMarginToPoint` (or a lock-aware wrapper around it)
knowledge of the excluded band, e.g. reject any case-(3) candidate whose
`[x0, point.x]` (or `[point.x, x0]`) span intersects `[bandLo, bandHi)`
before crediting it — mirroring the `arcIsClear` obstruction check the coin
model already uses in the same file for exactly this class of problem:

```js
// in checkKeyLockReachability, after computing `reach`:
const bandLo = lock.x, bandHi = lock.x + CONFIG.LOCK.W;
const reach = bestMarginToPointExcludingBand(
  { x: key.x, y: key.y }, cutNodes, spawnPaths, envelope, bandLo, bandHi,
);
```

or, more robustly, treat the key itself as a zero-width node that gets run
through the SAME `clipNodesAtBand`/`buildLockCutGraph` machinery already
proven sound for surface-to-surface connectivity, rather than through the
raw, cut-unaware point-reach math.

## Warnings

### WR-01: `keyHeld` is a single global boolean — no per-lock identity, unlike the validator's `keyId` model

**File:** `src/mechanics/key.js:47-120`, `src/scenes/game.js:66-72,304`
**Issue:** `scripts/lib/key-lock-check.mjs` explicitly models and self-tests
(case (d), lines 279-305) multiple *distinct* key-lock pairs disambiguated by
a `keyId` field on `geometry.locks`/`geometry.keys`. The actual in-engine
mechanic has no such concept: `wireKey`'s `hasKey()` callback
(`game.js:304`, `wireKey({ ..., hasKey: () => keyHeld })`) reads a single
scene-wide `keyHeld` boolean that is flipped `true` by picking up *any*
"key"-tagged entity (`key.js:58`, `onPickup()`), and every "lock"-tagged
entity's collide handler (`key.js:84`) opens on `hasKey()` alone — it never
checks which specific key was collected against which specific lock. A
future level authored with two independently-gated locks (which the
validator is designed to support and verify) would pass every static check
cleanly, but at runtime picking up *either* key would open *both* locks,
silently defeating the intended gating/puzzle structure. Not a softlock (the
opposite risk — it trivializes a lock), but a genuine functional mismatch
between the data model the validator enforces and what the engine actually
implements, currently invisible because level-02 ships exactly one pair.
**Fix:** Either thread key/lock identity through `wireKey` (track a
`Set<keyId>` of held keys instead of one boolean, and have the lock's
`hasKey` check test membership for its own `keyId`), or explicitly document
in `CONFIG.LOCK`/`key.js` that only ONE key-lock pair per level is supported
today and have `checkKeyLockReachability` warn/fail on `keyId`-disambiguated
levels until the engine side catches up.

### WR-02: level-02's key y-position is a bare magic number, not a named constant

**File:** `src/levels/level-02.js:110`
**Issue:** `keys: [{ x: 900, y: FLOOR_Y - 32 }]` — every sibling entity in
this same file (`goal`, `spikes`) derives its y from a named `CONFIG.*_SIZE`
constant (`FLOOR_Y - CONFIG.GOAL_SIZE`, `FLOOR_Y - CONFIG.SPIKE_SIZE`), but
the key uses a bare `32` that isn't `CONFIG.KEY.H` (20) — it happens to equal
the player's collider height (`PLAYER_H`, duplicated as a constant in
`scripts/lib/reachability.mjs:95`), which is presumably why it was chosen
(so the trigger's box aligns with a standing player's own AABB), but nothing
in `src/config.js` or this file documents that relationship, and the number
doesn't trace back to any single source of truth.
**Fix:** Either add a `CONFIG.KEY.STAND_Y_OFFSET` (or reuse a
player-height constant) and reference it here, or at minimum add an inline
comment explaining why `32` (not `CONFIG.KEY.H`) was chosen.

## Info

### IN-01: `key.js`'s `onPickup` docstring slightly overstates its call-once guarantee

**File:** `src/mechanics/key.js:44`
**Issue:** The JSDoc says `onPickup` is "called once, on the first key touched" — accurate for the single-key level-02 case, but for any future level with multiple key entities `onPickup` fires once per *distinct* key object touched (the `collected` Set is fire-once per key, not fire-once overall), so the wording could mislead a reader into assuming a single global latch.
**Fix:** Clarify to "called once per distinct key entity touched" or similar.
