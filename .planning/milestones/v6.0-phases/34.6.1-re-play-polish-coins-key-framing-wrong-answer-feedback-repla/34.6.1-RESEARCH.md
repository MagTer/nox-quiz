## §1 — No-Timer Anti-Mash Settle (D-01)

### The exact wrong-answer path today (`src/ui/challenge.js:289-296`)
```js
if (!correct) {
  audio.playSfx("wrong");
  if (box) box.color = rgb(CONFIG.PALETTE.DANGER[0], …);
  shake(6);
  return;                    // ← resolves INSTANTLY; mashing 1-2-3-4 slips the correct key through
}
```
`choose(i)` is the single funnel for BOTH `box.onClick(() => choose(i))` (line 259) and `keyCtrls = ["1".."4"].map((k,i) => onKeyPress(k, () => choose(i)))` (line 270). Guarding the TOP of `choose(i)` therefore covers both paths at once.

### Confirmed settle pattern (passes `check-safety.sh`)
1. **Closure-local flag** — declare `let settling = false;` in the `openChallenge` closure, right beside the existing `let cleared = false;` (line 97). Closure-local, never module-level — same anti-leak discipline as `cleared`/`keyCtrls`.
2. **Guard the funnel** — add `if (settling) return;` at the very top of `choose(i)` (line 278), after `if (cleared) return;` and before the bounds guard. Both mouse and keys go inert during the settle.
3. **Drive it off a flash tween's `.onEnd()`** — in the `if (!correct)` block:
```js
if (!correct) {
  settling = true;                                   // block input until the flash settles
  audio.playSfx("wrong");
  if (box) box.color = rgb(CONFIG.PALETTE.DANGER[0], CONFIG.PALETTE.DANGER[1], CONFIG.PALETTE.DANGER[2]);
  shake(CONFIG.GATE.WRONG_SHAKE);                    // stronger than the old shake(6)

  // Red pulse over the panel — a fixed() DANGER rect, tagged to THIS instance so
  // close() sweeps it if the session ever tears down mid-flash. One smooth easeOutQuad
  // fade (non-strobing, ADHD-safe), the SAME tween().onEnd() self-clean idiom as fx.js.
  const flash = add([
    rect(width(), height()),
    pos(0, 0),
    color(CONFIG.PALETTE.DANGER[0], CONFIG.PALETTE.DANGER[1], CONFIG.PALETTE.DANGER[2]),
    opacity(CONFIG.GATE.WRONG_FLASH_OPACITY),
    fixed(),
    z(CONFIG.GATE.WRONG_FLASH_Z),                     // above boxes (9993), brief pulse
    "challenge",
    instanceTag,
  ]);
  tween(
    CONFIG.GATE.WRONG_FLASH_OPACITY, 0,
    CONFIG.GATE.WRONG_SETTLE_MS / 1000,
    (v) => (flash.opacity = v),
    easings.easeOutQuad,
  ).onEnd(() => {
    settling = false;                                // re-enable input — same question stays open
    destroy(flash);
  });
  return;                                            // still forgiving: no close, no success, no lost state
}
```

This is the identical fire-once-cleared-by-`tween().onEnd()` shape as `key.js`'s `hintShowing` (`key.js:87,108-129`). `easings` and `tween`/`add`/`rect`/`destroy`/`width`/`height` are Kaplay globals already used inside this module's function bodies — a727c13-safe (they stay INSIDE `openChallenge`, never at module top level).

### check-safety.sh traps a naive implementation would trip (`scripts/check-safety.sh:59`)
The negative no-timer grep (comment-stripped) bans: `setTimeout | setInterval | countdown | [^a-zA-Z]wait\( | [^a-zA-Z]loop\( | lifespan\(`.
- **DO NOT** use `setTimeout`/`setInterval` to end the settle. ✗
- **DO NOT** use Kaplay `wait(0.25, …)`, `loop(…)`, or a `lifespan()` component to auto-destroy the flash. ✗ (`lifespan(` is explicitly banned; `wait(`/`loop(` are banned as call-forms.)
- **DO NOT** name any helper `wait(`/`loop(`/anything matching `countdown` — identifier names are scanned in code (only comments are stripped). Safe names: `settling`, `settleTween`, `flash`. ✓
- The forgiving grep (`gameOver|loseLife|penaliz|subtractXp|…`) is not at risk — the settle awards/subtracts nothing.
- `node --check` runs on every `src/*.js` first (line 51) — the edit must parse.

### Anti-softlock caveat (CONTEXT `<specifics>`)
Do NOT add a "refuse to answer" / "lock the wrong box permanently" path. The challenge must always remain resolvable — the settle only *delays* re-input by ~0.25s; the same question stays open and answerable. `settling` MUST be cleared by the tween `.onEnd()` (never left `true`), or the player is stranded.

### Tunables (all new, in `CONFIG.GATE`)
```js

WRONG_FLASH_OPACITY: 0.45,   // peak red-pulse opacity; single easeOutQuad fade (non-strobing)
WRONG_SHAKE: 8,              // shake magnitude on a wrong pick (replaces inline shake(6))
WRONG_FLASH_Z: 9995,         // above the answer boxes (z 9993), a brief pulse
```
(Flash color = `CONFIG.PALETTE.DANGER` = `[0xff,0x44,0x33]`, already defined.)

## §2 — Kaplay 3001 Sprite + Footprint-Preservation API

Confirmed against the vendored engine (`lib/kaplay.mjs`, v3001.0.19) and live in-tree usage.

### loadSprite with sliceX/anims (coin idiom, `main.js:78-85`)
```js
loadSprite("coin", "../assets/coin.png", {
  sliceX: CONFIG.COIN_FRAMES,                                   // horizontal cell count
  anims: { spin: { from: 0, to: CONFIG.COIN_FRAMES - 1, loop: true, speed: CONFIG.COIN_SPIN_SPEED } },
});
```
- `loadSpriteSheet` does NOT exist in Kaplay 3001 — `loadSprite` with `sliceX`/`sliceY`/`anims` is the only path (main.js:62 comment).
- Play the anim on the instance: `const coin = add([sprite("coin"), …]); coin.play("spin");` (`build.js:239-241`). Looping is set on the anim descriptor (`loop: true`), so `.play("spin")` runs forever.
- **Loader + sheet must agree.** If the vendored CC0 coin is NOT 8 frames of 32px, update `CONFIG.COIN_FRAMES` (and `COIN_SPIN_SPEED` if needed). A mismatch renders garbage frames. Measure the vendored sheet's real dimensions (`python3 -c "from PIL import Image; print(Image.open('assets/coin.png').size)"`) and derive `sliceX = width / frame_w`.

### area() collider derivation + footprint pinning
- **Default:** a bare `area()` on a sprite entity derives its collider from the sprite's **rendered** bounding box. A 16×16-rendered sprite → 16px collider; a 20×20-rendered sprite → 20px collider. This is why authoring/rendering at the collider size is the lowest-risk path.
- **Render at a fixed size:** `sprite("key", { width: CONFIG.KEY.W, height: CONFIG.KEY.H })` draws the sprite at exactly 20×20 regardless of the source PNG's native size (same `{ width, height }` mechanism `build.js` uses for terrain tiles, e.g. line 145). Combined with a bare `area()`, the collider is 20px — footprint preserved without an explicit shape.
- **Explicit shape (only if you need a visually-larger sprite over a small collider):** pin the collider independent of the visual, exactly as the spike does (`build.js:254-256`):
```js
area({ shape: new Rect(vec2(0), CONFIG.GOAL_SIZE, CONFIG.GOAL_SIZE), offset: vec2(0, 0) })
```
`Rect` is a Kaplay **class** global (not a factory) — `build.js:57` already guards `typeof Rect === "undefined"` inside the function body; any new `new Rect(...)` reuses that guarded scope.
- **Do NOT use `scale()` to shrink a big sprite** — `scale()` scales the WHOLE object including a bare `area()`, so the collider changes too. Use `sprite({ width, height })` (renders at target px, bare `area()` follows) OR an explicit `area({ shape })`.

**Recommended footprint-preservation choices for this phase:**
- **Goal (D-03):** author/vendor the checkered flag PNG to read at ~16×16 and keep the builder as-is (`sprite("goal")` + bare `area()` → unchanged 16px). If the chosen flag art is taller, render with `sprite("goal", { width: CONFIG.GOAL_SIZE, height: CONFIG.GOAL_SIZE })` + bare `area()`. A visually taller flag is allowed ONLY if the collider stays 16px and does not overlap platforms above (CONTEXT D-03) — verify per-level with `?debug=1`.
- **Key (D-04):** `sprite("key", { width: CONFIG.KEY.W, height: CONFIG.KEY.H })` + bare `area()` → unchanged 20px. `CONFIG.KEY.W`/`H` stay 20 (load-bearing).
