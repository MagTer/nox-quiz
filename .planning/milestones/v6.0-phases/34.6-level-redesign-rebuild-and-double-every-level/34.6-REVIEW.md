## Summary

Phase 34.6 makes the end-of-level gate key-conditional. The core refactor — extracting a shared `clearLevel({ table })` closure so the math-clear path and the key-skip path share one body for XP award, `markCleared`, HUD refresh, celebratory burst, `writeSave`, and the deferred `go("select")` transition — is sound and achieves its stated anti-drift goal. Those five effects genuinely cannot diverge between paths.

Verified clean:
- **No path divergence** in the extracted body: math path awards `progress.addXp(table)`, key path awards `progress.addBonusXp(CONFIG.PROGRESS.XP_KEY_SKIP)`; both then run identical `markCleared`/`hud.refresh`/`flashLevelUp`/`fx.clearBurst`/`writeSave`/transition code.
- **No double-clear / double-save race**: the `goalReached` fire-once latch (game.js:278-279) gates `onReachGoal`; the key branch `return`s before `openMathGate` (game.js:319), so `clearLevel` runs exactly once per goal.
- **No math gate after a key clear**: the `heldKeyIds.size > 0` branch returns early; `openMathGate` is unreachable on the key path.
- **No a727c13 violation**: all engine globals (`add`, `text`, `rect`, `color`, `tween`, etc.) remain inside the `gameScene()`/`clearLevel()`/`onReachGoal()` bodies; `config.js` additions are plain data.
- **No timers / no punishment constructs**: the transition still uses the `tween().onEnd()` self-clean idiom; the key path adds no `setTimeout`/`wait`/`loop`.
- **No run-state leak**: `heldKeyIds` and `KEY_DEFAULT_ID` are closure-local; `clearTransitionTween` is still cancelled in the `onSceneLeave` sweep.

Two warnings concern latent drift/robustness the extraction does *not* cover, plus one info-level test-coverage gap.

## Warnings

### WR-01: End-gate key-skip fires for ANY held key, conflating lock-keys with math-skip tokens

**File:** `src/scenes/game.js:293` (branch), `src/mechanics/key.js:89-104` (still-live LOCK half)
**Issue:** The end-gate skip triggers on `heldKeyIds.size > 0` — i.e. *any* key ever picked up in the run. But `wireKey` still fully supports the retired 34.5 LOCK half (a physical mid-level barrier opened by `hasKey(...)`), and both halves write/read the *same* `heldKeyIds` set. Nothing distinguishes a "math-skip token" from a "lock key". The moment a future level is authored with a mid-level lock+key pair (which the code path fully supports — `build.js:371-411` still emits `"lock"`/`"key"` and stashes `keyId`), collecting that lock-key to pass the wall would *also* silently skip that level's end math gate. That defeats the purpose of a lock as a mid-level barrier on an otherwise math-required level, and no validator catches it: `audit-endgate-key.mjs`'s discovery predicate (`keys.length > 0 && locks.length === 0`) deliberately *excludes* any lock-bearing level, so such a level would never be exercised by this gate at all. Currently latent (no shipped level has both), but it is a real trap given "double every level" will author more levels against a still-live lock mechanic.
**Fix:** Gate the end-skip on the token's intent, not mere possession. Either (a) branch on the level being a math-skip level (`level.geometry.keys?.length && !level.geometry.locks?.length`) before honoring the skip, or (b) tag skip-tokens distinctly (e.g. a `skipsEndGate` flag on the key descriptor, tracked in a separate set) so a lock-key never lands in the set the end-gate reads. Example (a):
```js
const isMathSkipLevel =
  (level.geometry.keys?.length ?? 0) > 0 && (level.geometry.locks?.length ?? 0) === 0;
if (isMathSkipLevel && heldKeyIds.size > 0) {
  // ...key-skip clear...
}
```

### WR-02: LEVEL CLEAR banner + clear SFX duplicated byte-for-byte across two files (with magic numbers), outside the anti-drift extraction

**File:** `src/scenes/game.js:298-316` duplicating `src/ui/mathGate.js:37-56`
**Issue:** The whole point of `clearLevel()` was "so they can never drift apart" — but the celebration banner + `audio.playSfx("clear")` are the one piece of the clear moment that was *not* pulled into the shared body. Instead the key branch re-implements mathGate.js's `onSuccess` banner inline, and the code comment itself admits it "mirrors src/ui/mathGate.js lines 37-56 byte-for-byte." Any future change to the banner (text, size, palette role, z-order, backdrop opacity) in one file silently diverges the two clear presentations. This also newly introduces magic-number literals into game.js that previously lived only in mathGate.js — confirmed unique-to-those-two-sites: `text("LEVEL CLEAR", { size: 30 })`, `z(9990)`, `z(9994)`, `color(0, 0, 0)` (grep shows game.js:304/308/313 now duplicate mathGate.js:43/47/52). The `config.js` `FX.BURST_Z` comment even documents `9990`/`9994` as if they were named constants ("above gate dim (9990), below LEVEL CLEAR banner (9994)"), yet they remain bare literals now copied into a second file, against the "no magic numbers in logic modules" binding rule.
**Fix:** Extract the banner+SFX into a single shared helper both paths call (e.g. `renderLevelClearBanner()` exported from mathGate.js, or a small function in game.js the mathGate `onSuccess` also invokes), and lift `size: 30`, `z: 9990`, `z: 9994` into `CONFIG.GATE`/`CONFIG.FX` keys referenced from the one helper. That folds the last un-shared piece of the clear moment into the same single-source discipline `clearLevel()` established for XP/save/transition.

## Info

### IN-01: audit-endgate-key.mjs asserts "free clear" but never verifies XP was awarded or persisted

**File:** `scripts/audit-endgate-key.mjs:359` (Path A `pathAHolds`)
**Issue:** The script's own header claims Path A proves the level "clears the level directly with full XP," but `pathAHolds = keyGone && preGoalChallengeCount === 0 && !goalTriggeredA && clearMarkerPresent` never inspects `localStorage`/`progress.xp` after the clear. A regression in the key branch's XP handling — `addBonusXp` not called, wrong `XP_KEY_SKIP` amount, or a `markCleared`/`writeSave` that silently no-ops on the key path — would pass this gate green. The "clear marker present (banner or on select scene)" check proves the transition ran, not that the save reflects the award.
**Fix:** After Path A settles, read the seeded save key back and assert the XP delta and the level's `cleared` fact, e.g.:
```js
const saved = await page.evaluate((k) => JSON.parse(localStorage.getItem(k) || "{}"), SAVE_KEY);
const xpAwarded = saved.xp >= XP_KEY_SKIP || saved.level > 1; // surplus carried or leveled
const clearedPersisted = saved.levels?.[levelId]?.cleared === true;
// fold both into pathAHolds
```
