    - `pink_fraction(...)` on an all-opaque, all-near-black (`(10,10,10)`) test image → returns `0.0` (near-black pixels have unstable/low saturation and must not spuriously match; `min_sat` correctly excludes them).

  <action>
    Create `scripts/lib/pink_scan.py` as a standalone Python module (no new dependency — Pillow only, matching `styleboard.py`/`scripts/build-art-assets.py`'s existing convention). Implement `pink_fraction(path, band_lo=211, band_hi=239, min_sat=30)`: open the file, `.convert("RGBA")`, get the alpha channel, then `.convert("RGB").convert("HSV")` (never convert a P-mode image straight to HSV — raises `ValueError` on Pillow ≥6.0, per this repo's own `hue_shift_band()` comment). Loop every pixel; skip where alpha is 0; among opaque pixels, count those with `saturation >= min_sat` AND `band_lo <= hue <= band_hi`; return `matched / opaque` (or `0.0` if `opaque == 0`).

    Lock these exact parameters as the module's defaults — validated during this phase's planning session (2026-07-10) against BOTH the real re-fetched Gothicvania pack files AND every currently-shipped `assets/` PNG: `band_lo=211`, `band_hi=239` (PIL's 0-255 hue units — narrower than a naive "200-255 pink range" specifically because the wider range false-positives on this project's existing shipped `goal.png`/`spike.png`, whose dominant colors sit at hue 244 and 185-188, just outside this narrower band), `min_sat=30`, and a trip THRESHOLD of `0.08` (8% of opaque pixels — the low end of CONTEXT.md's approved "~8-10%" range, chosen deliberately low-end so genuinely dominant-pink content like the Cemetery horizon glow still trips even after any brightness-based refinement).

    Add a CLI entry point (`if __name__ == "__main__":`) taking one path argument (a directory or single file). When given a directory, walk it recursively, SKIPPING any path component starting with `_` immediately after `assets/` (i.e. exclude `assets/_gothicvania-src/`, `assets/_kenney-src/`, `assets/_opengameart-src/`, `assets/_font-src/` — raw/pre-bake source, never the shipped scan target) and skipping non-`.png` files. For each remaining `.png`, compute `pink_fraction`; if it is `>= 0.08`, print the path and percentage to stderr and mark a failure, UNLESS the file's path (relative to repo root) is in a small, explicitly documented `ALLOWLIST` dict at the top of the module. Exit 1 if any non-allowlisted failure was found, else exit 0 and print a summary of any allowlisted files that were skipped (so the exception is always visible in the script's own output, never silent).

    Add exactly ONE allowlist entry now, following `scripts/check-rebrand.sh`'s documented-exception precedent (read above) — do NOT add more than this one entry speculatively: `"assets/player-swamphunter.png"`, justified in a comment citing the specific finding from this phase's planning session — the Swamp Hunter's native dark plum/maroon outline-shading color (sampled RGB `(50,34,46)`, HSV hue≈223/sat≈81/val≈50) numerically falls inside the pink/magenta hue band purely because hue is unstable at low brightness (a well-known HSV artifact for near-black pixel-art outline/shadow colors) — it is NOT a genuinely pink asset, and this exact sprite was the kid's explicit style-board pick (CONTEXT.md). If Plan 31-05's actual baked `assets/player-swamphunter.png` does not in fact trip the gate (e.g. because of resizing/cropping choices made there), this allowlist entry is harmless dead weight, not a bug — leave it in with its justification either way so the reasoning is preserved for future maintainers, and note its actual live/dead status in this plan's SUMMARY.md.

    Create `scripts/check-pink-gate.sh` as a thin wrapper matching `check-progress.sh`'s exact shape: `#!/usr/bin/env bash`, header comment explaining the gate's purpose and the locked ~8% threshold, `set -euo pipefail`, `ROOT="$(git rev-parse --show-toplevel)"`, a local `fail()` helper, then `python3 "$ROOT/scripts/lib/pink_scan.py" "$ROOT/assets" || fail "..."`, ending with `echo "pink-gate checks: PASS"` on success.
