This is the placeholder smoke test proving the vendored engine initializes; Phase 8 replaces it with real game code.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| upstream registry → repo | Vendored Kaplay file is downloaded once from a public registry and committed; after that there is no runtime fetch. |
| local file (file://) → browser | Accidental local-file open has a null/opaque origin where module/asset loading is blocked. |

(No backend, no database, no accounts, no user input, no secrets in this phase — the surface is a static asset set.)

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-SC | Tampering | Vendored `lib/kaplay.mjs` (supply chain) | mitigate | Pin exact version 3001.0.19; record source URL + sha256 in the header comment; download once from the official registry, never `npm install` at play time (zero runtime install vector). |
| T-07-01 | Denial of Service | `file://` accidental open | mitigate | Inline `file://` guard shows a clear "run via a web server" message instead of a silent blank canvas. |
| T-07-02 | Information disclosure | Repo internals leaking into served set | accept (this plan) | This plan only creates files; the image-copy scope (copy only `src/ lib/ assets/`, never `.git`/`archive/`) is enforced in plan 07-02's Dockerfile. |
</threat_model>
