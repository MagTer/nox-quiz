    Rebuild the keyboard cursor so it is row-aware, per CONTEXT.md's locked semantics:
    (a) Left/Right (`moveCursor(delta)`) must wrap the cursor WITHIN the current row ONLY -- never spill into an adjacent row. Compute the current tile's row from `Math.floor(selectable[cursor] / 4)`, find the subset of `selectable` cursor-indices whose tile also falls in that same row, and wrap `delta` within that subset (not the whole `selectable` array as today).
    (b) Add a NEW `moveCursorRow(delta)` for Up/Down: compute the current tile's row and column (`Math.floor(selectable[cursor] / 4)`, `selectable[cursor] % 4`), look at row `currentRow + delta`. If NO selectable tile exists in that target row, no-op (this is the "no cross-edge wrap" behavior -- pressing Down on the bottom row, or Up on the top row, does nothing). Otherwise land the cursor on the SAME column in the target row if it is selectable there; if not, land on the NEAREST selectable column in that row (smallest `|col - currentCol|`).
    (c) Wire `onKeyPress("up", () => moveCursorRow(-1))` and `onKeyPress("down", () => moveCursorRow(+1))` alongside the existing `onKeyPress("left", ...)`/`onKeyPress("right", ...)`/`onKeyPress("enter", ...)` registrations (lines 194-196) -- same registration style (plain app-bus controllers, auto-cleared by `go()`, no manual cancel needed, per this file's existing anti-leak comment).
    Do NOT change `paintCursor()`, the three-state tile palette, the mouse `onClick` wiring, or the `tiles`/`selectable` derivation itself -- this is a layout + cursor-stride generalization only.
