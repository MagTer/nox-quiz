## Tech Debt

### Phase 04 — Dungeon Renderer
- **Animation browser verification (human_needed):** Floating damage number lifecycle (animationend + 600ms fallback) and HP bar drain (300ms transition) require visual browser testing. Code is wired correctly; timing is unverifiable statically.

### Phase 05 — Full Floor Loop
- **DC-01 room-count spec mismatch (WARNING):** The implementation produces 4 regular combat rooms + 1 boss per floor (5 combat encounters total from ROOMS_PER_FLOOR=5). The requirement spec says "entrance → 3 combat rooms → boss room." Phase 5 VERIFICATION accepted the implementation as satisfying DC-01. Recommend updating the spec to reflect the actual design (dungeon-map = entrance, then 4 combats + boss) rather than changing working game logic.

### Phase 06 — Polish + ADHD Audit
- **SC-4 migration browser test (human_needed):** v1 localStorage migration requires manual verification with a real v1 save. 9-step test documented in Phase 6 VERIFICATION.md.
- **levelUpFlash 800ms (WARNING):** `@keyframes levelUpFlash` at line 274: `animation: levelUpFlash 0.8s`. Fires in dungeon mode via CombatInputHandler.handleAnswer result.leveledUp branch. ADHD-04 requires all combat animations ≤500ms. Phase 06 auditor passed this as "non-harsh celebratory flash" but the requirement is unqualified. Recommend reducing to 400–500ms.
